# 【深度篇】RAG 工程实践：从「能用」到「好用」

系列的第 11 篇讲了 RAG 的基础：索引、检索、生成三步，以及检索质量是能力上限。

但知道这些还不够。

真正做 RAG 的时候，你会遇到更具体的问题：用户问的问题措辞很随意，直接用来检索效果很差，怎么办？Chunk 切得太大太小都不对，到底怎么切？检索出来的结果顺序不对，模型用了排在后面的无关片段，怎么处理？RAG 改了一版，怎么知道是真的变好了？

这篇来回答这些问题。整体框架是 Advanced RAG 的三个阶段：**Pre-retrieval（检索前）、Retrieval（检索中）、Post-retrieval（检索后）**，加上评测体系。

---

## 一、Pre-retrieval：在检索之前做什么

检索质量很大程度上取决于检索之前的两件事：文档怎么处理，以及查询怎么改写。

### 1.1 文档解析与清洗

向量化之前，文档必须先被正确解析。解析质量差，后面所有环节都是在烂数据上做优化。

**不同格式的文档，解析难度差异很大：**

纯文本和 Markdown：最简单，基本没有解析问题。

PDF：是最常见也最难处理的格式。PDF 里的文字可能是扫描图片（需要 OCR）、可能有多列布局（解析顺序会乱）、可能有表格（表格内容容易被拆散）。常用工具：`pdfplumber`（表格提取好）、`pymupdf`（速度快）、`unstructured`（对复杂布局支持好）。

代码文件：不能按行切，要按语法单元切（函数、类），否则 Chunk 里的代码片段没有完整语义。用 AST 解析比纯文本切分好得多。

HTML / 网页：需要去掉导航栏、广告、页脚等噪音，只保留正文。`trafilatura` 和 `readability` 是常用的正文提取库。

**清洗的几个关键步骤：**

- 去除重复内容：同一段话出现在多个文档里，会稀释检索结果
- 去除低信息量内容：「版权所有」「本页空白」「第 X 页」这类内容不应进入索引
- 统一格式：标准化空白字符、编码，去除控制字符

**文档解析质量如何验证：** 随机抽样 50-100 个 Chunk，人工检查是否有乱码、截断、布局错乱。这一步很多人跳过，但它是发现解析问题最直接的方式。

### 1.2 Chunk 策略

文档解析完之后，需要切成 Chunk 再向量化。Chunk 策略对检索质量的影响，比大多数人想象的要大。

**固定大小切分（Fixed-size Chunking）**

最简单的方式：按 Token 数量切，每个 Chunk 固定大小，相邻 Chunk 有一定重叠（overlap）。

```python
# 示例：每个 Chunk 512 tokens，重叠 50 tokens
chunks = split_by_tokens(text, chunk_size=512, overlap=50)
```

优点：实现简单，适合结构均匀的文档。缺点：会在句子中间截断，破坏语义完整性。

**语义切分（Semantic Chunking）**

按语义边界切：句子、段落、章节。相邻句子之间计算语义相似度，相似度突变的地方就是切分点。

比固定大小切分效果好，但计算成本更高。适合对检索质量要求高的场景。

**递归切分（Recursive Chunking）**

按文档结构层级递归切：先按章节切，章节太长再按段落切，段落太长再按句子切。LangChain 的 `RecursiveCharacterTextSplitter` 就是这种策略。

对有明确层级结构的文档（技术文档、报告）效果好。

**专用切分策略**

- 代码：按函数/类切，保留完整语法单元
- 表格：整张表格作为一个 Chunk，或者把表格转成自然语言描述
- FAQ：问题和答案作为一个整体 Chunk，不要分开

**Chunk 大小怎么选：** 没有通用答案，取决于查询的粒度。如果用户的问题通常是具体的（「这个 API 的参数是什么」），小 Chunk（128-256 tokens）更精准；如果用户的问题需要综合多处信息（「这个系统的整体架构是什么」），大 Chunk（512-1024 tokens）更好。

建议的实验方法：用你的实际查询集，测试不同 Chunk 大小下的召回率，选召回率最高的配置。

### 1.3 Query 改写

这是 Pre-retrieval 里最有杠杆效应的优化，也是最容易被忽视的。

用户的原始查询通常不适合直接用于检索。原因有三：

**措辞随意**：「那个服务上周出的问题是啥来着」——自然语言的随意措辞，和文档里的规范表述往往不匹配。

**隐含上下文**：「它为什么会超时」——「它」指的是什么，只有在对话历史里才知道。直接用这个查询检索，什么都找不到。

**单一视角**：用户用一种方式问，但相关文档可能用完全不同的措辞描述同一件事。

**几种 Query 改写技术：**

**查询扩展（Query Expansion）**：用 LLM 生成原始查询的多个变体，分别检索，合并结果。

```
原始查询：「数据库连接超时」
扩展查询：
- 「database connection timeout error」
- 「连接池耗尽 connection pool exhausted」
- 「数据库连接失败 JDBC timeout」
```

**HyDE（Hypothetical Document Embeddings）**：用 LLM 生成一个「假设的理想答案文档」，用这个假设文档的向量去检索，而不是用查询的向量。原理是：答案文档的向量空间分布，和真实文档更接近，比查询向量更容易找到相关内容。

```
查询：「数据库连接超时怎么解决」
HyDE 生成的假设文档：
「数据库连接超时通常由连接池配置不当引起。
解决方案包括：1. 增大连接池大小... 2. 检查慢查询...」

用这段文字的向量去检索，而不是用原始查询
```

**查询分解（Query Decomposition）**：把复杂查询拆成多个子查询，分别检索，再合并。适合需要综合多处信息才能回答的问题。

```
复杂查询：「这次故障的根因是什么，以及如何防止下次发生」
拆解为：
- 「这次故障的根因」→ 检索故障记录
- 「如何防止类似故障」→ 检索最佳实践文档
```

**上下文补全（Contextual Query）**：结合对话历史，把指代词替换成具体实体。

```
对话历史：「我们在讨论 order-service 的数据库问题」
当前查询：「它的连接池配置是多少」
补全后：「order-service 的数据库连接池配置是多少」
```

## 二、Retrieval：检索阶段的优化

### 2.1 混合检索

正篇介绍了向量检索和关键字检索各自的优缺点。实践中两者结合（Hybrid Search）通常效果最好。

关键是**如何合并两路结果**。最常用的方法是 **RRF（Reciprocal Rank Fusion）**：

```python
def rrf_merge(keyword_results, vector_results, k=60):
    scores = {}
    for rank, doc in enumerate(keyword_results):
        scores[doc.id] = scores.get(doc.id, 0) + 1 / (k + rank + 1)
    for rank, doc in enumerate(vector_results):
        scores[doc.id] = scores.get(doc.id, 0) + 1 / (k + rank + 1)
    return sorted(scores.items(), key=lambda x: x[1], reverse=True)
```

RRF 不依赖两路检索的分数归一化（向量相似度和 BM25 分数的量级不同，直接加权很难调），只看排名，实现简单、效果稳定。

### 2.2 多粒度索引

单一粒度的 Chunk 有一个根本矛盾：Chunk 太小，语义不完整；Chunk 太大，噪音太多。

解决方案是**多粒度索引**：

**Parent-Child Chunk**：小 Chunk 用于检索（粒度细、精准），检索到小 Chunk 后，返回它所在的父 Chunk（更完整的上下文）给模型。

```
索引：把文档切成 128-token 的小 Chunk
检索：用小 Chunk 做相似度匹配
返回：匹配到的小 Chunk 所在的 512-token 父 Chunk
```

**句子窗口（Sentence Window）**：以句子为单位检索，返回时扩展到前后 N 句。

```
索引：每个句子单独向量化
检索：找到最相关的句子
返回：该句子 + 前后各 2 句（共 5 句的窗口）
```

两种方法都是「用小单元找精度，用大单元提供上下文」，是目前工程上效果最好的 Chunk 策略之一。

### 2.3 元数据增强

给每个 Chunk 附加结构化元数据，检索时可以用元数据做精确过滤，大幅提升精准度：

```json
{
  "content": "连接池大小为 20，QPS > 500 时曾触发连接耗尽",
  "metadata": {
    "source": "runbook/order-service.md",
    "section": "数据库配置",
    "doc_type": "runbook",
    "service": "order-service",
    "last_updated": "2024-01-15",
    "version": "v2.3"
  }
}
```

检索时可以先按元数据过滤（只检索 `service=order-service` 的文档），再做向量相似度匹配，大幅减少噪音。

**Contextual Retrieval（Anthropic 提出）**：在 Chunk 向量化之前，用 LLM 给每个 Chunk 生成一段上下文描述，描述这个 Chunk 在整个文档里的位置和作用，然后把描述和 Chunk 内容拼在一起再向量化。

```
原始 Chunk：「连接池大小为 20」

上下文描述（LLM 生成）：
「本段来自 order-service 运维手册的数据库配置章节，
描述了连接池的默认配置和历史故障案例。」

向量化内容：上下文描述 + 原始 Chunk
```

这样每个 Chunk 的向量里不只包含自身内容，还包含它在文档里的上下文语义，检索准确率会明显提升。

## 三、Post-retrieval：检索之后做什么

检索出来的结果，不能直接塞进 Prompt。还需要两件事：Reranking 和上下文压缩。

### 3.1 Reranking

向量检索的初始结果按相似度排序，但相似度高不等于相关性高——两者有差距。Reranking 用一个专门的 Cross-encoder 模型，对查询和每个 Chunk 做精细的相关性评分，重新排序。

```
向量检索初始结果（按相似度）：
1. Chunk A（相似度 0.92，但实际不相关）
2. Chunk B（相似度 0.88，真正相关）
3. Chunk C（相似度 0.85，相关）

Reranking 后：
1. Chunk B（相关性分 0.95）
2. Chunk C（相关性分 0.87）
3. Chunk A（相关性分 0.12，被排到后面）
```

常用 Reranking 模型：`bge-reranker`（开源，效果好）、`Cohere Rerank`（商业 API，效果优秀）。

Reranking 的代价是额外的延迟——对每个 Chunk 和查询做一次模型调用。实践中通常是：向量检索取 Top 20-50，Reranking 后取 Top 5-10 给模型。

### 3.2 上下文压缩

Reranking 之后，可能还有一些 Chunk 虽然整体相关，但只有其中一部分内容和当前查询直接相关。把整个 Chunk 都放进 Context，会引入不必要的噪音。

**LLM 压缩**：用 LLM 从每个 Chunk 里提取和查询最相关的句子或片段，丢弃无关内容。

```
查询：「order-service 连接池大小是多少」

原始 Chunk（500 tokens）：
「order-service 是订单系统的核心服务，负责处理用户下单请求。
该服务使用 HikariCP 连接池管理数据库连接，默认连接池大小为 20。
在 2024 年 1 月的一次故障中，由于连接池耗尽...（后续 400 tokens）」

压缩后（30 tokens）：
「order-service 使用 HikariCP 连接池，默认连接池大小为 20。」
```

**注意**：LLM 压缩会增加额外的延迟和成本，适合对答案精准度要求高、对延迟不敏感的场景。对延迟敏感的场景，可以用更简单的规则提取（关键句提取）替代。

## 四、RAG 的评测体系

RAG 改了一版，怎么知道是真的变好了？

RAG 的评测分两个层次：**检索质量**和**生成质量**，需要分别评测。

### 4.1 检索质量评测

**召回率（Recall@K）**：在 Top-K 个检索结果里，包含正确答案所在 Chunk 的比例。这是最直接的检索质量指标。

**MRR（Mean Reciprocal Rank）**：正确 Chunk 在检索结果里平均排在第几位。排得越靠前，MRR 越高。

**Context Precision**：检索到的 Chunk 里，真正相关的占多少。高 Precision 意味着噪音少。

评测数据集的构建：从知识库里抽样一批文档，手工（或用 LLM）生成对应的问题和「标准答案 Chunk」，构成 `(问题, 标准 Chunk)` 对。用这个数据集评测检索质量。

### 4.2 生成质量评测

**忠实度（Faithfulness）**：答案里的每一个陈述，是否都有检索到的文档支撑？不能出现幻觉——模型用了检索结果里没有的信息。

**答案相关性（Answer Relevance）**：生成的答案是否真正回答了用户的问题？

**上下文相关性（Context Relevance）**：检索到的上下文，和用户问题的相关程度。

这三个指标可以用 **LLM-as-Judge** 评测，也可以用专门的 RAG 评测框架：

**RAGAS**：专门为 RAG 设计的评测框架，同时评测 Faithfulness、Answer Relevance、Context Precision、Context Recall 四个维度，可以自动化运行。

**TruLens**：支持 RAG 评测和 Tracing，可以把评测结果和执行链路关联起来，定位是哪一步导致了质量下降。

### 4.3 端到端评测流程

```
1. 构建评测数据集
   ├── 从知识库抽样文档
   ├── 生成对应问题（手工 + LLM 辅助）
   └── 标注标准答案和标准 Chunk

2. 检索质量评测
   ├── 计算 Recall@K、MRR、Context Precision
   └── 定位召回率低的 case，分析原因（Chunk 策略？Query 改写？）

3. 生成质量评测
   ├── 用 RAGAS 计算 Faithfulness、Answer Relevance
   └── 人工抽查低分 case

4. 迭代优化
   ├── 检索质量问题 → 调整 Chunk 策略或 Query 改写
   ├── 生成质量问题 → 调整 Prompt 或 Post-retrieval 策略
   └── 每次修改后跑完整评测，确认没有退步
```

## 五、几个实践坑

**坑一：只优化向量模型，忽略 Chunk 策略**

换更好的向量模型，召回率提升有限；但把 Chunk 策略从「固定大小切分」改成「Parent-Child Chunk」，召回率可能有 10-20% 的提升。Chunk 策略的收益往往大于向量模型的收益，但花的时间远少于训练一个新模型。

**坑二：直接用原始查询做检索**

用户的自然语言查询，和知识库里的文档措辞往往不匹配。加上 Query 改写（哪怕只是简单的查询扩展），通常能显著提升召回率。这是成本最低、收益最高的优化之一。

**坑三：没有评测就改**

「感觉检索变好了」不是评测。每次修改 Chunk 策略、向量模型、Query 改写逻辑，都要跑一遍评测数据集，确认 Recall@K 有没有提升，有没有引入退步。没有评测的优化是盲目的。

**坑四：Reranking 模型没有校准**

直接用开源 Reranking 模型，不做任何校准就上线。Reranking 模型的效果依赖于它的训练数据分布，和你的业务文档分布可能不匹配。至少要用你的评测数据集验证 Reranking 是否真的提升了 MRR，而不是盲目信任模型的通用基准分数。

**坑五：知识库更新了，索引没有同步**

文档更新了，但向量索引还是旧的——这会导致 RAG 返回过时的信息。需要设计增量更新机制：文档变更时触发对应 Chunk 的重新向量化，而不是定期全量重建（全量重建成本高，而且有时间窗口内的数据不一致）。

## 最后说一句

RAG 不是「装个向量数据库，把文档切碎塞进去」就完了。

从文档解析到 Chunk 策略，从 Query 改写到 Reranking，从元数据过滤到上下文压缩——每一个环节都有优化空间，每一个环节做差了都会成为瓶颈。

但这也意味着：一个精心设计的 RAG 系统，和一个随手搭的 RAG 系统，检索质量可能有天壤之别。差距不在模型，在工程。

把评测体系建起来，用数据驱动每一次优化决策——这是 RAG 工程化最重要的一条原则。
