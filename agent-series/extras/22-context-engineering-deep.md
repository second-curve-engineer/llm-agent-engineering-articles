# 【深度篇】Context Engineering：从「知道概念」到「能落地」

系列的第 02 篇讲了 Context Engineering 的基础：Context 会爆炸、要精准放内容、不是越多越好。

但知道这些还不够。

真正做 Agent 的时候，你会遇到更具体的问题：Context 该在什么时机压缩？压缩的摘要质量怎么保证？关键信息放在 Prompt 的什么位置影响最大？不同类型的任务，Context 策略应该有什么区别？Context 策略改了，怎么知道是变好了还是变差了？

这篇来回答这些问题。

---

## 一、Lost in the Middle：位置比内容更重要

先讲一个很多人不知道的现象。

2023 年 Stanford 的一项研究（后于 2024 年发表于 TACL）发现：当关键信息位于长 Context 的中间位置时，模型在部分问答和检索任务上的表现会下降；位于开头或结尾时通常表现更好。这个现象被称为 **Lost in the Middle**。

实际含义是：**你把关键信息放在 Context 的什么位置，直接影响模型能不能用上它。**

对 Agent 来说，这意味着几件具体的事：

**稳定的系统规则放在高优先级位置**——System Prompt 的位置和优先级通常由 API 消息协议决定（system / developer 角色），行为约束、工具规范、输出格式，要放在这里，并与动态任务内容明确分隔。

**最新的、最相关的信息放在最后面**——用户的当前问题、最新的工具返回结果、最近一轮的推理结论，都应该尽量靠近 Context 的末尾。

**不重要的中间内容做压缩**——早期轮次的详细过程、已经确认不相关的工具返回、冗余的推理步骤，可以在中间位置做摘要甚至删除，腾出空间给更重要的信息。

一个可供验证的经验策略是：**高优先级规则保持稳定，最新任务和关键证据靠近当前请求，低价值历史则进行压缩或外置。** 但不同模型、消息协议和任务类型的长上下文表现不同，最终效果仍需通过 Eval 验证。

## 二、什么时候压缩，怎么压缩

压缩是 Context 管理最核心的工程问题。压缩太早，丢了有用信息；压缩太晚，Context 已经爆炸。

**触发压缩的时机**

不要等到 Context 超限再压缩——那时候已经被动了。工程上通常设置一个低于窗口上限的**软阈值**，在达到阈值时开始压缩早期内容。具体比例需要结合预留输出空间、工具定义大小、任务类型和 Eval 结果确定——60-70% 可以作为初始实验值，而不是固定标准。

具体触发条件：
- Token 数超过软阈值（比如 128K 窗口设 80K 触发）
- Loop 轮次超过某个数（比如每 5 轮做一次压缩）
- 某个阶段性目标完成（比如完成了「信息收集」阶段，进入「分析」阶段）

**摘要的质量是关键**

压缩本质上是用摘要替代原始内容。摘要的质量直接决定了压缩后 Agent 能不能继续正常工作。

摘要质量差的表现：
- 摘要太短，丢失了关键细节（「查了日志，发现了问题」）
- 摘要太模糊，没有具体数据（「响应时间有些长」而不是「P99 延迟 3.2s，超出阈值 2x」）
- 摘要遗漏了关键的否定信息（「已排除了网络问题」这类结论一旦丢失，后面可能重复排查）

做好摘要需要一个专门的摘要 Prompt，明确要求：保留具体数字、保留已排除的假设、保留当前的推理状态。

一个摘要 Prompt 的示例框架：

```
将以下执行历史压缩成摘要，要求：
1. 保留与后续决策有关的数字、时间、标识符和证据，并记录来源
2. 明确标注已排除的可能性
3. 说明当前已确认的事实
4. 说明当前推理方向和下一步计划
5. 过滤无关细节，脱敏敏感内容
格式：已知事实（含来源）/ 已排除 / 当前方向
```

**滑动窗口 vs 摘要替换**

压缩有两种基本策略：

**滑动窗口**：保留最近 N 轮的完整历史，更早的轮次直接丢弃。实现简单，但会硬性丢失信息——如果早期某个关键发现在 N 轮之外，就找不回来了。适合任务链条短、早期信息对后期影响小的场景。

**摘要替换**：把早期 N 轮的详细内容替换成一段摘要，保留关键结论，丢弃原始过程。保留的信息更完整，但依赖摘要质量。适合长任务、早期信息对后期推理仍有价值的场景。

实践中通常组合使用：先做摘要替换，当摘要积累得太多时，对摘要再做二次压缩。

## 三、不同任务类型，Context 策略不同

不是所有 Agent 任务都适用同一套 Context 策略。任务类型不同，对 Context 的需求差异很大。

**故障排查类任务**

特点：多步推理，早期证据对最终结论至关重要，信息有强时序依赖。

策略：优先用摘要替换而非滑动窗口；摘要里必须保留所有已排除的假设。关键证据及其来源必须保留在 Context 中；大体积原始工具返回结果可存入外部存储，在 Context 中保存摘要、引用 ID 和必要片段，需要时再取回。

**代码生成/修改类任务**

特点：需要大量代码上下文，但大部分代码是静态的（不会随执行变化），对话历史相对不重要。

策略：代码上下文应按需获取，可以结合文本搜索、符号索引、依赖分析、repo map 和语义检索，而不是全量塞入 Context；只保留最近的修改记录和用户的最新需求；错误信息和测试结果完整保留。

**多轮对话类任务**

特点：用户意图会在对话中不断细化，早期的用户表述可能被后期覆盖。

策略：维护一个「用户意图摘要」，随对话滚动更新；不保留早期的完整对话，只保留意图演变的关键节点。

**长时间运行的任务**

特点：可能跑几十轮甚至上百轮，随着轮次增加，Context 的体积、信息冲突、过期内容和压缩误差都会持续累积，管理难度显著上升。

策略：引入分阶段的 Context 管理——每个阶段（信息收集、假设验证、结论生成）维护自己的 Context 快照，阶段切换时做完整摘要，之前阶段的详细内容全部归档到外部存储。

## 四、Working Memory：一个被低估的设计

在 Context 里单独维护一块 **Working Memory**，是 Context Engineering 里最有杠杆效应的设计之一。

Working Memory 是一个持续更新的任务状态摘要，格式固定，内容精简。每一轮执行结束后更新，始终放在 Context 的固定位置（通常是 System Prompt 之后、执行历史之前）。

一个故障排查 Agent 的 Working Memory 示例：

```
[Working Memory - 第 6 轮更新]

已确认事实：
- 故障发生时间：2024-01-15 14:23
- 受影响服务：order-service
- 错误类型：数据库连接超时（timeout > 30s）
- 并发量：故障期间 QPS 正常，无流量异常

已排除：
- 网络问题（ping 延迟正常）
- 应用代码变更（最近 48h 无发布）

当前方向：
- 数据库连接池配置异常（连接池耗尽的可能性高）
- 待查：数据库慢查询日志
```

Working Memory 的价值：

**防止「失忆」**：即使早期的详细执行历史被压缩，Working Memory 里的关键状态始终清晰呈现，模型不会在后续轮次里「忘记」前面已经确认的事情。

**加速推理**：模型每轮推理时，先看 Working Memory 知道现在在哪、还需要做什么，不需要从头扫描整个历史。

**辅助快速复盘**：Working Memory 可以帮助快速了解任务进展，但它是有损的状态摘要，不能替代 Trace。原始工具调用、模型输出和状态变更仍需单独记录在 Trace 里。

Working Memory 的设计要点：
- 格式要固定，不要让模型自由发挥
- 内容要结构化（已确认/已排除/当前方向），不要用自然语言叙述
- 每条事实应附带来源 ID（如工具调用 ID 或轮次编号），避免摘要把模型推测写成已确认事实
- 在产生新事实、排除假设、计划变化或阶段切换时更新；没有有效变化时可以跳过。更新最好采用字段级合并，而不是每轮全文重写，避免反复改写产生漂移

## 五、怎么评测 Context 策略的效果

Context 策略改了，怎么知道是变好了还是变差了？这是很多人没想清楚的问题。

**不要用「感觉」判断**

「感觉这次结论更准了」——这不是评测，这是自我安慰。Context 策略的效果评测必须量化。

**三个可量化的指标**

**任务完成率**：在一组标准测试用例上，修改 Context 策略前后的任务成功率变化。这是最直接的指标。

**效率指标**：在任务成功率不下降的前提下，平均轮次、Token 消耗、延迟和工具调用数量下降，才说明策略效率可能提高。单看轮次减少不够——轮次少也可能是过早结束或漏掉了必要的检查步骤。

**关键信息召回率**：在需要用到早期信息的测试用例上，模型有没有正确引用了关键结论？（比如「已排除网络问题」这个结论，在后续轮次里有没有被正确利用）。这需要人工或 LLM-as-Judge 评测。

**建立 Context 专项测试集**

Context 策略的测试用例，应该专门设计来暴露 Context 问题：
- **长任务用例**：必须跑 10 轮以上才能完成的任务，用来测压缩策略
- **早期关键信息用例**：第 1-2 轮收集的信息，在第 8-10 轮必须被正确引用
- **否定信息用例**：测试摘要是否保留了「已排除」的内容
- **信息冲突用例**：工具返回了互相矛盾的结果，测试模型能不能正确处理

## 六、几个实践坑

**坑一：把所有工具返回结果都保留**

「工具返回的数据说不定后面有用」——这种想法会导致 Context 快速膨胀。正确做法是：工具返回结果进 Context 之前，先提炼出「这次工具调用发现了什么」写进 Working Memory，原始数据可以只保留最近 2-3 轮，更早的归档到外部存储。

**坑二：摘要 Prompt 没有专门设计**

用通用的「请总结以上内容」来做 Context 压缩，摘要质量会非常不稳定。摘要 Prompt 需要针对任务类型专门设计，明确要保留什么格式、什么信息。

**坑三：压缩时机太晚**

等到接近 Context 上限才开始压缩，导致压缩时可操作的空间很小，摘要质量急剧下降。设置软阈值，在 60-70% 时触发，给压缩过程留足空间。

**坑四：忽略 Context 里的噪音**

Context 里的信息不只是「多少」的问题，还有「质量」的问题。一段解析失败的工具返回（乱码、空结果、错误堆栈）如果不加处理地塞进 Context，会稀释模型的注意力。工具返回结果进 Context 之前，要做基本的质量过滤。

**坑五：对所有任务用同一套 Context 策略**

Context 策略不是通用的，不同任务类型需要不同的策略。花时间分析你的 Agent 主要处理什么类型的任务，针对性地设计 Context 管理，收益远高于试图找到一个「万能策略」。

**坑六：忽略 Prompt Cache 友好性**

很多 LLM API 支持 Prompt Cache——如果请求的前缀和上一次相同，可以跳过重新处理，降低延迟和成本。但如果每轮都修改 Context 的前半部分（比如频繁更新 System Prompt 里的工具列表），Cache 会频繁失效。稳定的系统规则和工具定义应放在 Context 前缀的固定位置，动态变化的内容（Working Memory、最新证据）放在后部，这样前缀保持稳定，Cache 命中率更高。

---

## 最后说一句

Context Engineering 是 Agent 工程里最难量化、最难调试、也最容易被忽视的部分。

它不像 Guardrail 那样「加上就能看到效果」，也不像 Eval 那样「有个分数就知道好坏」。Context 策略的影响是弥散的——它渗透在 Agent 每一轮的推理质量里，出了问题很难直接归因。

但这也意味着：把 Context 做好的 Agent，和把 Context 随便处理的 Agent，在复杂任务上的表现差距可能是本质性的，而不只是微小的分数提升。

这是 Agent 工程里最值得投入的地方之一。
