# Agent 工程深度篇：面试 Q&A

> 使用说明：每道题先自己想答案，再看要点。要点是关键词和核心逻辑，不是完整回答。

---

## 一、State Management（状态管理）

**Q1：Agent 在 HITL 等待期间，为什么不能让进程一直挂着？怎么解决？**

要点：
- 进程挂着占用资源，服务可能重启/宕机，审批回来时进程已不存在
- 解决：状态持久化到外部存储（DB/Redis），进程可以释放
- Resume 时从存储读取状态，重建 Agent 执行上下文

---

**Q2：Agent 的 State 你会分成哪几类，各自存在哪里？**

要点：
- 执行状态（当前轮次/进度/等待事件）→ 关系型 DB，需要事务
- 上下文状态（消息历史/可重建 Context）→ 热冷分层，Redis + 对象存储
- 证据状态（工具返回/推理结论）→ KV 存储，有语义检索需求用向量存储
- 任务状态（任务元信息/status）→ 关系型 DB
- 关键点：选型看一致性/访问模式/数据大小/恢复延迟，没有固定答案

---

**Q3：Checkpoint 和工具调用之间有个原子性缺口，怎么理解，怎么解决？**

要点：
- 问题：工具已执行有副作用，Checkpoint 还没写入时崩溃 → 恢复后重复执行
- 仅靠 Checkpoint + 幂等 ID 不能保证 exactly-once
- 解决：执行前记录 `pending`，完成后写结果；无法幂等的操作用状态查询或补偿
- 进阶：transactional outbox 模式确保状态写入和副作用的原子性

---

**Q4：Resume 的目标是让模型输出和之前完全一样吗？**

要点：
- 不是，模型有非确定性，追求完全相同不现实
- 目标是：关键事实/工具结果/计划状态/约束一致
- 动态内容（当前时间/实时数据）在 Checkpoint 时存快照，恢复时用快照

---

**Q5：Multi-Agent 场景下，多个 Sub-Agent 并发写状态，怎么保证一致性？**

要点：
- 分区隔离：每个 Sub-Agent 只写自己的分区，汇总时处理冲突
- 乐观锁：写入时带版本号，冲突时重试
- 消息队列：按任务 ID 分区，单写入者顺序消费；仍需处理重复消息/乱序
- 每条状态要有归属标记（agent_id/timestamp/version/source）

---

**Q6：Agent 系统升级了新版本，旧 Checkpoint 怎么处理？**

要点：
- Checkpoint 里要存 schema_version/agent_version/prompt_version/model_config
- 版本不兼容的处理：向前兼容（自动补默认值）/ 状态迁移（迁移脚本）/ 降级终止
- 版本检查要在 Resume 最开始做，不能等到执行中才发现

---

## 二、Context Engineering（上下文工程）

**Q7：什么是 Lost in the Middle，对 Context 设计有什么启示？**

要点：
- 研究发现：关键信息在长 Context 中间时，模型在部分问答和检索任务上表现下降
- 启示：高优先级规则放稳定位置，最新事实靠近当前请求，低价值历史压缩或外置
- 不是通用定律，效果因模型/任务/协议而异，需 Eval 验证

---

**Q8：Context 压缩的时机怎么选？压缩摘要质量怎么保证？**

要点：
- 时机：设软阈值（比如 60-70% 作为初始值），不要等到超限才压缩；阶段性目标完成时也可触发
- 摘要质量：用专门设计的摘要 Prompt，明确要求保留：与后续决策相关的数字/时间/来源、已排除的假设、当前推理方向；过滤无关细节，脱敏敏感内容
- 滑动窗口 vs 摘要替换：前者简单但硬丢信息，后者保留关键结论但依赖摘要质量

---

**Q9：Working Memory 是什么，怎么设计，和 Trace 有什么区别？**

要点：
- Working Memory：任务级别的结构化状态摘要，格式固定，放在 Context 固定位置
- 三块：已确认事实（附来源 ID）/ 已排除假设 / 当前方向
- 更新时机：产生新事实/排除假设/计划变化时更新，用字段级合并不要全文重写
- 和 Trace 的区别：Working Memory 是有损摘要，Trace 是可审计的原始记录，不能互相替代

---

**Q10：不同任务类型的 Context 策略有什么区别？**

要点：
- 故障排查：用摘要替换，保留已排除假设，关键证据附来源，大体积结果存外部存储
- 代码生成：代码用按需获取（文本搜索/符号索引/语义检索），只保留最近修改记录
- 多轮对话：维护「用户意图摘要」滚动更新，不保留早期完整对话
- 长时间任务：分阶段管理，阶段切换做完整摘要，旧阶段归档外部存储
- 核心：没有万能策略，根据任务特征设计，用 Eval 验证

---

**Q11：Prompt Cache 友好设计是什么意思，怎么实现？**

要点：
- 很多 LLM API 支持 Prompt Cache：前缀相同可以跳过重新处理，降低延迟和成本
- 问题：频繁修改 Context 前半部分（如工具列表）会导致 Cache 失效
- 实现：稳定的系统规则和工具定义放 Context 前缀固定位置，动态变化内容（Working Memory/最新证据）放后部

---

## 三、Memory（记忆系统）

**Q12：什么信息值得写入长期记忆，什么不值得？写入时机怎么选？**

要点：
- 值得写：跨任务复用价值高 / 纠正了错误的先验认知 / 用户明确且适合复用的偏好（按作用域保存）
- 不值得写：只在当前任务有意义的中间结果 / 原始工具数据 / 已有更权威来源的信息
- 写入时机：未验证推断不实时写入；用户明确声明/工具验证的关键事实可即时写入；其余内容任务结束后由「记忆提炼」步骤集中处理

---

**Q13：记忆的数据结构应该包含哪些字段，为什么？**

要点：
- source_type（tool_result / user_stated / model_inferred）：来源不同，可信度不同
- verification_status（verified / unverified / disputed）：和来源是不同维度
- confidence_score（0-1）：数值型置信度
- valid_from / valid_until：时效控制，不靠 last_used_at 猜测
- sensitivity（public / internal / confidential）：权限控制
- supersedes：版本链，指向被取代的旧记忆 ID
- tags + tenant_id / user_id / project_id：作用域和检索过滤

---

**Q14：新信息和旧记忆冲突了怎么处理？**

要点：
- 不要直接覆盖，用版本化更新，通过 supersedes 链接
- 根据来源权威性决定：权威工具结果可以生成新版本；来源接近/证据不足时标记 disputed
- disputed 的记忆等后续工具验证或人工确认，不静默覆盖
- 冲突检测：写入前检查同 subject 的记忆，也可以用语义相似度检测

---

**Q15：记忆检索为什么不能「先检索再过滤」？正确的做法是什么？**

要点：
- 先检索再过滤：向量相似度计算已经访问了所有用户数据，信息已经泄露；topK 结果不准；资源浪费
- 正确做法：权限条件下推到存储查询层，作为查询的一部分执行
- 实现方案：向量数据库元数据过滤（Qdrant/Weaviate/Pinecone 支持）/ 命名空间物理隔离 / 数据库 RLS
- 多作用域：并行查询私有/项目/公共记忆，合并后 Reranking
- 向量本身也是敏感数据，高敏感场景用物理隔离

---

**Q16：什么是 Memory Poisoning，怎么防？**

要点：
- 攻击者通过构造特殊输入，把恶意指令植入长期记忆，影响 Agent 未来行为
- 防御：写入长期记忆的内容来源必须可信；user_stated 和 model_inferred 使用时有更高审查门槛，不能和 tool_result 同等信任
- 外部文档/工具返回不直接转写为长期记忆，需经过「记忆提炼」步骤过滤和验证

---

## 四、RAG（检索增强生成）

**Q17：HyDE 是什么原理，有什么风险？**

要点：
- 原理：用 LLM 生成「假设的理想答案文档」，用其向量检索，而非原始查询向量
- 原理：答案文档向量空间和真实文档更接近，比查询向量更容易找到相关内容
- 风险：假设文档中的错误概念可能导致检索偏移，需通过评测决定是否启用

---

**Q18：Parent-Child Chunk 和 Sentence Window 是什么，解决什么问题？**

要点：
- 核心矛盾：Chunk 小→语义不完整；Chunk 大→噪音多
- Parent-Child：小 Chunk 做检索（精准），返回时给父 Chunk（完整上下文）
- Sentence Window：句子粒度检索，返回时扩展到前后 N 句
- 本质：用小单元找精度，用大单元提供上下文
- 是解决粒度与完整性矛盾的常见方案，效果需在自己数据集验证

---

**Q19：RRF 是什么，为什么用 RRF 合并混合检索结果，而不是直接加权？**

要点：
- RRF（Reciprocal Rank Fusion）：基于排名而非分数合并，每个结果的得分 = Σ(1/(k+rank))
- 为什么不直接加权：向量相似度和 BM25 分数量级不同，归一化困难，调参复杂
- RRF 只看排名，不依赖分数归一化，实现简单、效果稳定

---

**Q20：RAG 的评测体系怎么设计，Recall@K 和 MRR 分别是什么？**

要点：
- 两个层次：检索质量（Recall@K / MRR / Context Precision）和生成质量（Faithfulness / Answer Relevance）
- Recall@K：Top-K 结果覆盖了多少相关 Chunk，衡量覆盖度
- MRR：每个查询第一个相关结果排名倒数的平均值，越接近 1 说明相关结果越靠前
- 评测数据集：从知识库抽样文档，生成 (问题, 标准 Chunk) 对
- 工具：RAGAS（自动化检索+生成评测）/ TruLens（评测+Trace 关联）
- 关键原则：每次改动后跑完整评测，用数据驱动优化决策

---

**Q21：Contextual Retrieval 是什么，和普通 RAG 有什么区别？**

要点：
- Anthropic 提出：向量化前，用 LLM 给每个 Chunk 生成上下文描述（在文档中的位置和作用）
- 描述和 Chunk 内容拼在一起再向量化，上下文也可加入 BM25 索引
- 区别：普通 RAG 的向量只包含 Chunk 自身内容；Contextual Retrieval 的向量还包含上下文语义
- 实际提升幅度需在自己数据集验证，不能直接套用公开实验数字

---

## 五、Multi-Agent（多智能体）

**Q22：什么时候应该用 Multi-Agent，什么时候不该用？**

要点：
- 该用：子任务之间真正独立可并行 / 超出单 Agent Context 承载 / 有明确专业化边界
- 不该用：子任务强依赖必须串行 / 协调成本超过并行收益 / 还没有 Trace/Eval/状态管理基础设施
- 原则：Multi-Agent 的收益来自合理分工，不来自 Agent 数量本身

---

**Q23：静态 Orchestrator 和动态 Orchestrator 有什么区别，各自适合什么场景？**

要点：
- 静态：任务拆解逻辑写死在代码里，行为可预期、容易测试
- 动态：用 LLM 做任务拆解，灵活但不可预期，难以测试和排查
- 建议：除非任务多样性真正要求动态拆解，否则优先用静态；可预期的系统比灵活的系统更容易维护

---

**Q24：Sub-Agent 之间的消息格式应该包含哪些字段，为什么？**

要点：
- session_id：把整个 Multi-Agent 任务的所有消息关联起来，可观测性基础
- correlation_id：根任务 ID，所有子任务都带上，便于追溯完整链路
- message_type：task_request / task_result / task_error，接收方按类型路由
- constraints（含 timeout）：每个子任务必须有超时，不能无限等待
- confidence：Sub-Agent 对结果的置信度，Orchestrator 合并时可加权
- token_usage：监控各 Sub-Agent 成本

---

**Q25：某个 Sub-Agent 失败了，有哪些处理策略，怎么选？**

要点：
- 四种策略：部分结果继续（非关键 Sub-Agent）/ 结果替代（降级用能力相似的）/ HITL（关键 Sub-Agent 失败宁可等待）/ 任务终止（后续步骤无法继续）
- 选择依据：失败的 Sub-Agent 是不是关键路径上的 / 结果质量是否达到最低要求 / 任务的风险容忍度
- 重试设计：次数限制（2-3次）+ 指数退避 + 幂等前提 + Sub-Agent 级别而非任务级别

---

**Q26：Multi-Agent 的 Trace 怎么设计才能串联所有 Sub-Agent 的执行链路？**

要点：
- 根任务启动时生成 root_run_id，所有 Sub-Agent 的 Trace 都携带这个 ID
- Trace 结构：root_run_id → Orchestrator → Sub-Agent A / Sub-Agent B / Sub-Agent C
- 用 root_run_id 可以拉出整个 Multi-Agent 任务的完整链路
- 关键监控：端到端成功率 / Sub-Agent 失败率 / 降级触发率 / 各 Sub-Agent 的延迟和 Token 消耗
- 调试技巧：先隔离测试每个 Sub-Agent / 录制回放 / 先串行再并行

---

## 综合题（高频考察）

**Q27：你怎么评估一个 Agent 系统的整体质量？**

要点：
- 不能只看最终答案对不对，需要分层评测
- 单步质量（工具调用参数/格式）/ 轨迹质量（路径合理/轮次/成本）/ 端到端质量（答案准确/完整）
- 量化指标：任务成功率 / 平均轮次（需在成功率不下降前提下看）/ Token 成本 / 是否触发必要 HITL
- 建立回归测试集，每次迭代跑完整 Eval，用数据驱动决策

---

**Q28：你会怎么排查一个 Agent 系统出现的"越用越差"问题？**

要点：
- 先确认问题范围：是所有任务都差，还是某类任务变差？是最近一次改动引入的？
- 检查 Context：Context 是否膨胀（Working Memory 是否失控，是否有大量无用历史）
- 检查 Memory：长期记忆是否被错误信息污染（Memory Poisoning？记忆冲突没处理？）
- 检查工具：工具返回质量是否下降（外部服务变化？数据 schema 变化？）
- 检查模型：模型是否有版本变更
- 工具：用 Trace 定位具体哪轮开始跑偏，用 Eval 对比前后版本的指标变化

---

**Q29：让你从零设计一个企业级 Agent 系统的记忆模块，你会怎么做？**

要点：
- 首先明确需求：什么类型的记忆（用户偏好/系统知识/任务历史）/ 作用域（tenant/用户/项目）/ 安全要求
- 数据结构：source_type / verification_status / valid_from|until / sensitivity / supersedes / tags
- 写入：记忆提炼步骤（任务结束后）/ 冲突检测和版本化更新 / 防 Memory Poisoning
- 检索：权限过滤下推到存储层（元数据过滤/RLS）/ 混合检索 + Reranking / 多作用域并行查询
- 生命周期：按记忆类型设验证周期 / 主动验证机制 / 完整删除支持（含索引/缓存/派生数据）
- Eval：建立专项测试用例（任务 N 写入，任务 N+K 正确使用）

---

**Q30：面试官问你「Agent 和普通后端服务有什么不同，你的后端经验怎么迁移」**

要点：
- 相同点：API 设计→Tool Schema / 日志监控→Trace / 测试→Eval / 权限系统→Tool Permission+Guardrail / 异常处理→HITL / 状态机→Agent Runtime
- 核心不同：输出是概率性的（不是确定性的），需要评测而不只是断言；调用链是 LLM 推理驱动的，不是代码控制流
- 后端经验最有价值的地方：稳定性思维（重试/幂等/降级）/ 可观测性（Trace/监控）/ 状态管理 / 权限设计
- 需要补的新能力：Context Engineering / Prompt 设计 / LLM 调用特性 / Eval 体系
