# Agent 的状态管理：持久化、Resume 和一致性

Agent 跑到第 7 轮，触发了一个高风险操作，进入 HITL 等待人工审批。

审批可能等 10 分钟，也可能等 8 小时。进程不能一直占着资源等待，而且服务可能重启、机器可能宕机，审批回来时原来的执行进程可能已经不存在。

那 Agent 的状态怎么办？

如果没有设计好状态管理，答案只有一个：任务失败，从头重跑。但从头重跑意味着前 7 轮的工具调用、证据、模型输出和任务进展可能全部丢失。这不只是浪费，有些工具调用有副作用——已经发出去的请求、已经修改的数据，重跑一遍可能造成重复执行。

这就是 Agent 状态管理要解决的核心问题：**让 Agent 在任意时刻都能安全暂停，在恢复时从断点精准继续，而不是从头重来。**

---

## 一、Agent 的 State 是什么

在讨论怎么管理之前，先搞清楚 State 的边界。

Agent 运行时产生的数据，可以分成四类：

**执行状态（Execution State）**
当前跑到第几轮、任务整体进展如何、是否在等待某个事件（HITL 审批、外部回调）、终止条件是否已满足。这是 Agent Runtime 控制循环所需要的最核心状态，没有它就不知道下一步该做什么。

**上下文状态（Context State）**
每一轮调用 LLM 所需的输入来源——System Prompt、执行历史、工具定义、当前任务描述。Resume 的基础不是保存每轮完整 Context，而是保存足以重建后续输入的消息、状态摘要、工具结果和必要快照，恢复时据此重建，而不是原样回放。

**证据状态（Evidence State）**
Agent 在执行过程中收集到的所有中间结果——工具返回值、LLM 推理结论、已确认的事实、已排除的假设。这是任务的"工作记忆"，直接影响后续的推理质量。

**任务状态（Task State）**
当前任务的元信息——任务 ID、创建时间、发起人、优先级、超时设置、当前状态（running / pending_hitl / completed / failed）。这是任务调度和监控所需要的外层状态。

这四类状态的变化频率不同：执行状态每轮都在变，上下文状态每轮追加，证据状态按需更新，任务状态只在关键节点变化。理解这个区别，对存储设计很重要。

## 二、持久化：怎么把状态存下来

持久化要解决的问题：**Agent 随时可能中断，状态必须在中断前完整保存，恢复时能还原到中断前的精确状态。**

**序列化**

State 是运行时的内存对象，持久化的第一步是序列化——把内存里的状态转成可以存储的格式（JSON、MessagePack、Protobuf 等）。

序列化的难点不在于技术，而在于**边界**：哪些东西该序列化、哪些不该。工具的实例对象不能序列化，但工具的调用记录（名称、参数、返回值）需要保存。LLM 的连接不能序列化，但需要保存可恢复、可审计的关键输入输出；敏感数据和超大结果则应脱敏、引用外部对象或按策略裁剪，不要无差别全量落盘。

一个实用原则：**序列化的是数据，不是对象。** 恢复时用序列化的数据重新构造对象，而不是试图直接恢复对象本身。

**存储位置**

存储选型主要看一致性需求、访问模式、数据大小、恢复延迟和保留周期。结构化状态通常需要事务能力；大对象可放对象存储；Redis 更适合缓存和短期热状态，而不是唯一的持久化来源。以下是常见的参考方向，具体选型应结合实际约束：

- **执行状态 + 任务状态**：需要事务性更新和按任务 ID 查询，通常选关系型或文档数据库。
- **上下文历史**：读写频繁、数据量增长快，可考虑热冷分层，热数据缓存、冷数据归档。
- **证据状态**：Key-Value 按需读写，有语义检索需求时考虑向量存储。

**检查点（Checkpoint）**

不是每一步都需要持久化——频繁写入会拖慢性能。工程上通常在以下时机做 Checkpoint：

- 每轮 Loop 结束后
- 触发 HITL 之前（必须持久化，否则恢复不了）
- 任务状态发生变化时

需要特别注意的是：**「工具调用完成后再写 Checkpoint」存在原子性缺口**——工具已经产生副作用，但 Checkpoint 尚未写入时崩溃，恢复后仍可能重新触发工具调用，造成重复执行。仅靠 Checkpoint 无法实现 exactly-once 语义。

对有副作用的工具，需要额外的幂等机制：在执行前记录 `pending` 状态，完成后写入结果；Runtime 检测到 `pending` 时判断是否需要查询结果或补偿，而不是直接重试。必要时可使用 transactional outbox 模式确保状态写入和副作用的原子性。

Checkpoint 是 Resume 的前提：没有 Checkpoint，就没有可以恢复的断点。

## 三、Resume：从断点精准恢复

持久化解决了"存下来"的问题，Resume 解决"恢复回去"的问题。

**精准 Resume 的难点**

简单的 Resume 是：把序列化的状态读出来，重新构造 Agent 对象，从上次的断点继续执行。

但有几个细节容易出问题：

**工具调用的幂等性**：如果 Agent 在工具调用之后、写入 Checkpoint 之前崩溃了，恢复时不知道这个工具调用有没有成功执行。Runtime 为每次调用生成幂等键，工具执行层持久化该键与结果；重复请求到来时直接返回已有结果，而不是重新执行。对于无法实现幂等的操作（如发送通知、外部转账），则需要状态查询或补偿机制，而不能只依赖幂等键。

**Context 的重建**：Resume 时需要重建足以支撑后续推理的输入。目标不是让模型再次生成完全相同的输出（模型本身具有非确定性），而是保证关键事实、工具结果、计划状态和约束一致。对于动态内容（比如当前时间、实时数据），需要在 Checkpoint 时存储快照，恢复时使用快照而不是重新获取，避免语义漂移。

**HITL 的 Resume**：HITL 是最常见的暂停场景。人工审批完成后，需要把审批结果（批准/拒绝/修改参数）注入到 Agent 的状态里，然后从暂停的那一步继续。注意：人工可能修改了参数，Resume 时需要用新参数覆盖原来的工具调用请求，而不是用原来的参数重新执行。

**Resume 的验证**

Resume 之后的第一件事，应该是验证状态的完整性：
- 执行状态是否完整（轮次、进度）
- 证据是否完整（没有遗漏的工具返回结果）
- Context 历史是否一致（每轮的输入输出都在）

验证失败，比盲目继续执行要好——至少可以及时发现状态损坏，而不是在错误状态上继续跑出错误结论。

## 四、Multi-Agent 下的状态一致性

单个 Agent 的状态管理相对简单，因为只有一个执行者。Multi-Agent 场景下，多个 Sub-Agent 并发读写共享状态，一致性问题就来了。

**共享状态的并发问题**

典型场景：Orchestrator 把任务拆给三个 Sub-Agent 并行执行，三个 Sub-Agent 都需要读写同一个 Evidence Store。如果两个 Sub-Agent 同时写入，可能产生覆盖或冲突。

常见的处理方式：

**分区隔离**：每个 Sub-Agent 只写自己的状态分区，Orchestrator 负责在汇总时合并。写入不冲突，合并时处理冲突。适合任务可以明确分区的场景。

**乐观锁**：写入时带上版本号，如果发现版本冲突（另一个 Sub-Agent 已经写入了更新的版本），重试或放弃。适合冲突概率低的场景。

**消息队列**：可以按任务 ID 分区，由单写入者顺序消费状态事件，减少并发冲突。但消息队列本身仍需处理重复消息、乱序投递和消费失败，不能直接等同于串行化写入。适合对一致性要求高、可以接受额外复杂度的场景。

**状态的归属**

Multi-Agent 里，每条状态都应该有明确的归属：谁写的、写入时间、来自哪个 Sub-Agent。这不只是为了一致性，也是为了 Trace 和复盘——出了问题能追溯到具体是哪个 Sub-Agent 的哪一步写入了错误的状态。

建议每条状态记录至少包含：`agent_id`、`run_id`、`timestamp`、`version`、`source`（哪个工具或 LLM 调用产生的）。

## 五、状态的生命周期管理

状态不是永久存储的，需要有明确的生命周期策略：

**保留策略**：任务完成后，状态保留多久？保留和删除策略应根据恢复窗口、审计要求、数据敏感度、成本及隐私法规确定，不能使用统一期限。有些合规场景要求长期保留，有些隐私法规（如 GDPR）反而要求在特定条件下彻底删除。

**清理策略**：过期状态怎么清理？通常先逻辑删除（标记为 archived），确认无需恢复后再物理清理。但对于有隐私删除要求的场景，需要支持真正的物理删除，而不是只做软删除。

**容量控制**：Context 历史会随着 Loop 轮次增长，单个任务的状态大小需要有上限。超过上限时，早期的详细记录可以被摘要替代——保留关键结论，丢弃原始数据。

**审计要求**：在需要合规的场景，状态记录不能被修改，只能追加。每一次状态变更都要有 immutable log，可以重现任意时刻的状态。

## 六、状态版本与迁移

Agent 部署新版本后，旧 Checkpoint 里的状态可能已经不兼容——Schema 变了、Prompt 改了、工具参数调整了、模型换了。如果直接用旧状态恢复，可能产生难以排查的运行时错误，甚至静默跑出错误结论。

工程上需要在状态里显式记录版本信息：

```json
{
  "schema_version": "2.1.0",
  "agent_version": "1.4.2",
  "prompt_version": "prod-2024-01-15",
  "tool_versions": {
    "query_logs": "3.0.1",
    "check_metrics": "2.1.0"
  },
  "model_config": {
    "model": "claude-3-5-sonnet",
    "temperature": 0.0
  }
}
```

**版本不兼容时的处理策略**：

**向前兼容**：新版本能读取旧格式的状态，自动补齐缺失字段的默认值。适合小版本迭代、变化不大的情况。

**状态迁移**：提供迁移脚本，把旧格式的 Checkpoint 转换成新格式，再 Resume。适合 Schema 有结构性变化的情况。

**降级终止**：检测到版本不兼容时，拒绝 Resume，把任务标记为需要人工处理。适合变化太大、迁移成本太高或安全要求严格的情况。

**版本检查应该在 Resume 最开始做**——在读取任何业务状态之前，先校验版本兼容性。发现不兼容，立刻走降级路径，而不是在执行过程中才暴露问题。

## 七、工程实践中的常见坑

**坑一：状态和业务逻辑耦合**
把 Agent 的状态直接绑定在业务对象上（比如直接修改数据库里的业务记录），导致状态回滚和业务回滚纠缠在一起。好的做法是 Agent 状态和业务状态严格分离，Agent 状态只记录 Agent 的执行过程，业务状态由工具层负责。

**坑二：没有考虑工具调用的幂等性**
Resume 时盲目重试所有工具调用，导致有副作用的操作被重复执行。幂等性设计应该从工具定义阶段就开始考虑，而不是出了问题再补救。

**坑三：Context 重建不完整**
Resume 时只恢复了执行状态，没有完整重建 Context 历史，导致 LLM 在后续轮次"失忆"——不知道前面已经发现了什么，开始重复之前的工作。

**坑四：Multi-Agent 状态没有归属标记**
所有 Sub-Agent 往同一个 Evidence Store 写入，没有 `agent_id` 标记，出了问题无法追溯是哪个 Sub-Agent 写入了错误数据。

**坑五：状态大小没有上限**
长时间运行的 Agent 任务，Context 历史和 Evidence Store 会持续增长，最终超出存储限制或序列化性能瓶颈。需要在设计阶段就定好容量上限和压缩策略。

---

## 最后说一句

State 管理是 Agent 工程化里最容易被低估的部分。

早期写一个 Demo，不需要考虑 State——任务跑完就完了，出了问题重跑一遍。但一旦 Agent 需要处理长时间任务、需要 HITL、需要在生产环境稳定运行，State 管理就变成了绕不过去的工程问题。

把状态设计好，Agent 才能真正做到：任意时刻可暂停，任意断点可恢复，任意错误可追溯。

这不是高级特性，而是生产可用的基础条件。
