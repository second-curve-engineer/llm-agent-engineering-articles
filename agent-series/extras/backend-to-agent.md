# 我拆开一个 Agent 系统之后，发现它还是个后端项目

最近两年，我身边有一种很普遍的焦虑。

做后端的朋友开始担心：CRUD 岗位越来越少，招聘市场上"Agent 工程师"、"AI 应用工程师"的 JD 越来越多。很多人的第一反应是——我是不是要转行？要去学大模型？要去研究 Prompt？

我也有过这种感觉。

直到我认真拆了几个 Agent 系统之后，发现了一件让我有点意外的事：

**把 LLM 从 Agent 系统里拿掉，剩下的几乎全是后端问题。**

---

## 一、先承认现实：传统岗位确实在变

我不想说"后端工程师没什么好怕的"，因为这不诚实。

AI 工具确实在提升开发效率。重复 CRUD、模板页面、简单接口——这些工作越来越容易被 AI 辅助生成。市场对"纯执行型工程师"的需求，确实在被压缩。

但这里有一个容易被忽略的区分：

**被压缩的，不是"后端工程"，而是"只会按需求写接口"。**

这两件事不是同一件事。

---

## 二、我拆开一个 Agent 系统，看到了什么

一个完整的 Agent 系统，大概长这样：

```
用户输入
  ↓
Agent Runtime（控制循环）
  ↓
Router / Planner（决策层）
  ↓
Tool Registry（工具注册与调度）
  ↓
Memory / Evidence Store（状态与上下文）
  ↓
外部系统（数据库、API、文件）
  ↓
LLM（推理层）
  ↓
输出 / 下一轮
```

乍一看，LLM 在里面，感觉很"AI"。

但如果我把 LLM 这一层单独拿出来，剩下的是什么？

- **Agent Runtime**：管理任务生命周期，控制执行循环，处理中断和恢复——这是任务调度系统。
- **Router / Planner**：根据输入决定走哪条路径，调用哪个工具——这是业务流程编排。
- **Tool Registry**：管理工具的注册、权限、调用接口——这是 API 设计 + 权限系统。
- **Memory / Evidence Store**：存储上下文、中间结果、历史状态——这是缓存 + 状态管理。
- **Trace / Observability**：记录每一步的输入输出、耗时、异常——这是日志监控。
- **HITL（人工介入）**：遇到高风险操作暂停，等人确认再继续——这是审批流 + 异常处理。
- **Eval**：用可量化的标准衡量输出质量，跑回归——这是测试体系。

我数了一下：七个核心模块，六个半是后端工程问题。

## 三、能力映射：你已经会的，在 Agent 里叫什么

这是我觉得最有价值的一张表。

很多后端工程师焦虑，是因为看到了一堆新词——Context Engineering、Tool Calling、Agent Runtime——觉得自己什么都不会。

但如果把新词翻译回来：

| 传统后端能力 | Agent 工程中的对应物 |
|------------|-------------------|
| API 设计 | Tool Schema / Function Interface |
| 微服务调用链 | Tool Calling / Multi-Agent 协作 |
| 业务流程编排 | Workflow / Router / Planner |
| 日志与监控 | Trace / Observability |
| 异常处理与重试 | Retry / Fallback / Self-Correction |
| 人工审批流 | HITL（Human-in-the-Loop）|
| 权限系统 | Tool Permission / Data Boundary |
| 数据库与缓存 | Memory / Evidence Store / RAG |
| 测试与回归 | Eval / Scenario Tests |
| 稳定性治理 | Agent Runtime 可靠性 / Guardrail |

你会发现：**换了名字，没换掉本质。**

## 四、真正变化的是什么

但我也不想说"Agent 跟后端一模一样，没什么新东西"，因为这同样不诚实。

真正发生变化的，是一件更深层的事：

**软件工程的抽象层，正在上移。**

以前，工程师的核心工作是：直接实现业务逻辑。`if-else`、数据库操作、接口返回——逻辑由人写，确定性执行。

现在，越来越多的业务逻辑开始交给 LLM 推理。工程师的工作变成了：**管理一个会推理的系统**。

具体来说：

- 以前你写 `if 错误类型 == A, then 执行操作 B`
- 现在你设计 Context，让 LLM 判断当前是不是 A 类错误，该不该执行 B

逻辑还在，但实现方式变了。工程师从"写逻辑"变成了"设计让 LLM 能正确推理的环境"。

这是真实的变化。但注意：**变化的是工程对象，不是工程能力的价值。**

---

## 五、什么样的后端工程师会被淘汰，什么样的不会

这个问题值得直说。

**更容易被替代的：**
只做需求翻译和 CRUD，不理解系统复杂度，不关心稳定性、权限、可观测性。这类工作的确越来越容易被 AI 工具和低成本人力替代。

**反而更有价值的：**
能理解复杂系统、稳定性、状态管理、权限边界、故障排查的工程师——这些能力在 Agent 时代不仅没有贬值，反而更稀缺。

因为 Agent 系统最大的难点，从来不是"调一下 LLM API"。而是：

- 怎么让它稳定跑在生产环境
- 出了问题怎么排查
- 高风险操作怎么控制
- 质量怎么量化和保证

这些问题，有后端系统经验的工程师，反而更有直觉。

---

## 六、需要补的新能力

当然，光靠老经验不够。Agent 工程有它自己的新东西需要学。

需要补的：
- **LLM 调用方式**：怎么构造 Prompt，理解 Token、温度、上下文窗口
- **Tool Calling**：怎么设计工具接口，让 LLM 能正确调用
- **Context Engineering**：怎么管理上下文，控制信息的放入和截断
- **Trace / Eval**：怎么让 Agent 的执行过程可见、质量可量化
- **Agent Runtime 设计**：怎么控制执行循环、状态持久化、精准恢复

但底层的工程判断不能丢：
- 可控性
- 可观测性
- 可测试性
- 权限和安全边界
- 成本和性能

这些是后端工程师天然的底色，在 Agent 时代仍然是核心竞争力。

---

## 最后说一句

后端工程师这个 title 可能会变，但后端工程能力不会消失。

它会迁移到 Agent Runtime、工具系统、企业级 AI 工程化里。换一批新名词，解决一批新问题，但工程的本质没有变：**让系统稳定、可控、可观测地运行。**

真正需要升级的，不是简历上的名称，而是对工程对象的理解。

AI 时代的软件工程师，本质上仍然是软件工程师。只是，我们现在要管理的系统，会推理了。

这篇是「工程师的第二曲线」Agent 系列的收官篇。

最后这篇，我想说的是：这些东西，你其实并不陌生。

写这个系列的初衷，不是为了介绍一堆新概念，而是想说清楚一件事——**Agent 工程化，本质上是软件工程在 AI 时代的延伸**。如果你有后端系统的经验，你已经站在一个很好的起点上了。

接下来怎么走，每个人的路不一样。但我相信，把工程做扎实的人，在哪个时代都不会太差。
