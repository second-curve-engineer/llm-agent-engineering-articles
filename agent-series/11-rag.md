# Agent 的外部记忆：一文讲透 RAG

Agent 接到一个任务：根据公司内部的故障处理手册，给出这次数据库超时的处理建议。

它开始推理，给出了一份看起来很专业的建议——但你翻了一下手册，发现它的建议和公司规范对不上，有几条关键步骤完全没提到。

问题出在哪？

Agent 用的是模型训练时学到的通用知识，而不是你公司内部那份手册。那份手册在模型训练截止之后才写的，或者根本就没有进入训练数据。

模型不知道它不知道什么。它会用已有的知识自信地回答，即使答案是错的。

这就是没有 RAG 的 Agent：知识被锁在训练截止日期里，无法获取外部的、私有的、最新的信息。

---

## 一、为什么 Agent 需要 RAG

模型的知识有三个天然的局限：

**知识有截止日期**：模型训练完成之后，世界还在变。新的故障案例、新的规范文档、最新的系统状态——这些内容不会自动进入模型参数；如果没有外部检索或工具，模型就无法基于它们回答。

**无法获取私有数据**：公司内部的运维手册、历史故障案例库、私有代码库——这些数据不会出现在公开训练集里。模型对它们一无所知。

**上下文窗口有限**：即使你把所有文档都塞进 Prompt，Context 窗口也装不下。何况很多知识库有几十万甚至几百万字，根本不可能全量放进去。

RAG（Retrieval-Augmented Generation）解决的就是这三个问题：**在模型推理的时候，动态检索相关内容，注入到 Context 里，让模型基于真实、最新、私有的知识来回答。**

---

## 二、RAG 是什么

RAG 的核心思路很简单：与其让模型"记住"所有知识，不如在需要的时候"查"到相关知识，再交给模型推理。

入门版 RAG 可以理解成三步：

**索引（Indexing）**：把知识库里的文档切成小块（Chunk），用向量模型把每个 Chunk 转成向量，存入向量数据库。这是离线完成的预处理步骤。

**检索（Retrieval）**：Agent 有了问题，把问题也转成向量，在向量数据库里找和问题最相似的几个 Chunk，取回原文。

**生成（Generation）**：把检索到的 Chunk 和原始问题一起放进 Prompt，让 LLM 基于这些内容生成答案。

RAG 的名字来自这三步：检索（Retrieval）增强（Augmented）生成（Generation）。检索是手段，增强生成质量是目的。

生产系统里通常还会加上：文档解析与清洗、权限过滤、Query Rewrite（查询改写）、Reranking（二次排序）、引用生成和 Eval。每一环都会影响最终效果。

---

## 三、检索质量为什么是能力上限

对于依赖外部知识的问题，检索结果决定了模型能看到哪些证据。检索到了相关内容，模型有机会给出准确答案；检索没找到，或者找回来的是不相关的内容，模型只能靠自己的训练知识猜——而这正是我们想避免的。

检索质量往往决定 RAG 的证据上限：如果关键证据没被召回，再强的模型也很难答对。但召回之后，模型能否正确使用证据、克制地引用而不乱发挥，同样重要。

检索质量主要受几件事影响：文档解析质量、Chunk 策略、Embedding 模型、查询改写（Query Rewrite）、Metadata 过滤、混合检索策略和 Reranking。

**文档解析**：PDF、表格、代码、图片——不同格式的文档解析质量差异很大。解析出的文本质量直接影响后续所有步骤。

**Chunk 策略**：文档怎么切块，决定了检索的粒度。切太大，一个 Chunk 包含多个主题，检索时信噪比低；切太小，关键信息被割裂，检索到的内容缺乏上下文。好的 Chunk 策略需要根据文档类型和查询特征来设计，不是一刀切。

**向量模型的质量**：向量模型决定了语义相似度的计算精度。用通用向量模型处理专业领域文档，语义理解会有偏差——"数据库超时"和"慢查询"在通用语义空间里可能不够近，但在数据库运维语境里显然相关。

**查询改写（Query Rewrite）**：用户的原始问题往往不适合直接用于检索。让 Agent 先把问题改写成更适合检索的形式，或者拆成多个子查询，能显著提升召回率。

**检索策略**：纯向量检索（语义匹配）擅长找语义相关的内容，但对精确关键字匹配效果差；纯关键字检索（BM25）擅长精确匹配，但理解不了语义。在很多文本知识库场景里，混合检索（Hybrid Search）结合两者，再加上 Reranking 模型对结果二次排序，是常见且效果稳定的工程组合。

## 四、RAG 在 Agent 里怎么用

在 Agent 系统里，RAG 常被封装成一个检索工具或检索服务，负责「给定问题，返回相关文档片段」。Agent 把它当成工具来调用——和调用数据库查询、调用 API 一样，RAG 是 Agent 工具箱里的知识检索工具。

```
工具名：search_knowledge_base
输入：用户问题或 Agent 的子查询
输出：最相关的 N 个文档片段 + 来源引用
```

Agent 在推理过程中，判断当前问题需要外部知识时，主动调用这个工具，把检索结果注入到后续的推理上下文里。

这里有一个设计细节值得注意：**查询的质量影响检索结果**。直接把用户的原始问题丢给检索，不如让 Agent 先把问题拆解成更具体的子查询，再分别检索。"排查数据库超时"这个问题，拆成"数据库超时的常见原因"、"慢查询的排查步骤"、"索引优化方法"三个子查询，检索召回率会明显更高。

**权限过滤**

企业 RAG 还有一个容易被忽视的问题：权限。不是检索到了就能给模型看。不同用户能访问的文档不同，检索层必须做权限过滤，否则 Agent 可能把不该看的内部资料带进 Context，形成数据泄露。权限过滤通常在检索之后、注入 Context 之前完成，是企业级 RAG 系统的必要组件。

## 五、工业界怎么做

**LlamaIndex**

专注于 RAG 和知识检索的框架，提供从文档解析、Chunk 切分、向量化、检索到 Reranking 的完整流水线。支持多种向量数据库和检索策略，对复杂文档结构（表格、PDF、代码）的处理能力较强。是很多团队做 RAG 工程化时会重点考虑的框架之一。

**LangChain**

提供完整的 RAG 组件，包括文档加载器、文本分割器、向量存储接口和检索链。和 LangGraph 的 Agent 框架深度集成，适合已经在 LangChain 技术栈上的团队。

**向量数据库**

RAG 的底层存储。主流选项：
- **Pinecone**：全托管，上手快，适合不想自维护基础设施的团队
- **Weaviate**：开源可自部署，支持混合检索（向量 + BM25）
- **Qdrant**：开源，性能好，支持过滤和混合检索
- **pgvector**：PostgreSQL 插件，适合已经用 PG 的团队，降低基础设施复杂度

**Reranking 模型**

检索之后的二次排序，显著提升最终召回质量。常用方案：Cohere Rerank、BGE Reranker、cross-encoder 类模型。代价是增加一次模型调用的延迟和成本，但在对准确率要求高的场景下值得。

---

## 六、没有 RAG 的代价

没有 RAG 的 Agent，在通用问答上可能表现不错，一旦涉及私有知识或时效性信息就暴露问题：

**知识幻觉**：模型用训练知识自信地回答，但答案可能已经过时或者根本不适用于你的场景。用户很难区分模型是真的知道，还是在"编"。

**无法使用私有数据**：公司内部文档、历史案例、专有知识——这些最有价值的信息，模型永远访问不到。

**答案不可溯源**：模型给出的答案来自训练数据，无法告诉你依据是哪份文档的哪一段。设计得好的 RAG 系统可以为答案附上来源引用，让可信度和可审计性都更高。

有了 RAG，Agent 的知识边界才从"训练截止日期"扩展到"你的知识库有多新、有多全"。

---

## 最后说一句

RAG 是 Agent 的外部记忆。

模型的参数记忆是静态的，训练完就固定了；RAG 让 Agent 有了动态获取知识的能力——知识库更新了，Agent 的回答就更新了，不需要重新训练模型。

但 RAG 不是银弹。它解决的是"知识从哪来"的问题，解决不了"推理对不对"的问题。检索到了对的内容，模型还是可能推理出错；检索没找到，再强的模型也没有用。

**把检索做好，是 RAG 工程化最核心的投入。**
