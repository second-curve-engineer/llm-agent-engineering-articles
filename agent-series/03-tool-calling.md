# Agent 的手：一文讲透 Tool Calling

前两篇讲了 Agent Loop 和 Context Engineering。

有人可能会问：Loop 每一轮，Agent 到底是怎么"干活"的？它怎么查日志、读文件、调接口？

答案是 Tool Calling。

## 一、LLM 其实什么都做不了

先说一个很多人没意识到的事实。

如果只看裸模型的一次调用，它的能力边界其实很窄：没有网络权限，没有文件系统，也不会天然知道当前时间。它本质上是在根据输入生成输出。

但 Cursor 能帮你改代码，故障排查 Agent 能帮你查日志、分析报错、定位根因。

它们是怎么突破这个限制的？

靠的就是 Tool Calling——LLM 和外部世界之间的接口协议。

## 二、Tool Calling 的底层机制

很多人以为 Agent 调用工具，是 LLM 直接执行了某段代码。

实际上完全不是。

完整的流程分四步：

**第一步：告诉 LLM 它有哪些工具**

开发者会把工具的名称、功能、参数格式注册给模型——有些系统写在提示词里，有些系统通过专门的 tools schema 作为独立参数传给模型。LLM 在推理时，知道自己"有哪些手可以用"。

**第二步：LLM 输出一份"申请单"**

LLM 推理之后，不会直接执行任何操作。它会输出一个结构化的 tool call 请求——你可以把它理解成一份 JSON 申请单：

类似这样：

```json
{
  "tool": "query_log",
  "parameters": {
    "trace_id": "abc-123",
    "time_range": "last_1h"
  }
}
```

**第三步：外部系统真正去执行**

这段 JSON 被外部的 Agent Runtime 解析，找到对应的工具函数，用 LLM 填的参数真正去执行——查数据库、读文件、发 HTTP 请求……

**第四步：结果塞回 Context，LLM 继续推理**

执行结果作为 Observation 写回 Context，LLM 拿到结果，开始下一轮推理。

> 📌 【插图二：Tool Calling 四步流程图】

**关键认知：LLM 从来不直接执行任何操作，它只是在"写申请单"。** 真正动手的，始终是外部系统。

这个设计不是偶然的——它让 LLM 专注推理，让执行层专注可靠性，两者职责分离。

这里其实有三层：**LLM 负责决定意图，Tool Calling 协议负责表达意图，Agent Runtime 负责真正执行。** 下一篇会专门拆 Runtime 这一层。


## 三、LLM 怎么"决定"用哪个工具

这是很多人好奇但很少文章讲清楚的问题。

主要靠**语义理解**——LLM 会读懂每个工具的描述，结合当前任务判断哪个工具最合适、参数填什么。但成熟的系统通常还会叠加规则、权限和路由策略，共同决定最终走哪个工具。

这意味着，**工具描述写得好不好，直接影响 Agent 的行为质量。**

来看一个对比：

**差的工具描述：**
```
工具名：query_log
描述：查询日志
参数：id, time
```

**好的工具描述：**
```
工具名：query_log
描述：根据 trace_id 查询指定时间范围内的服务日志，
      返回日志级别、时间戳和错误信息。适用于排查
      接口报错、异常堆栈、超时问题。
参数：
  - trace_id（必填）：请求链路 ID
  - time_range（必填）：查询范围，如 last_1h / last_24h
  - level（可选）：日志级别过滤，如 ERROR / WARN
```

> 📌 【插图三：好工具描述 vs 坏工具描述对比图】

差的描述让 LLM 猜；好的描述让 LLM 精准决策。写好工具描述，是 Agent 工程里最容易被忽视、但回报很高的一件事。


## 四、工具调用失败了怎么办

真实工程里，工具不会每次都成功。

**工具返回错误**
LLM 拿到报错信息，重新推理，换一个参数或换一个工具再试。这是 Agent 自我修正能力的来源。但不是所有工具都应该自动重试——对有副作用的操作（发邮件、写数据库、调支付接口），Runtime 通常要加权限确认、幂等控制和重试限制，否则一次失败可能触发多次执行。

**参数格式填错**
LLM 有时会生成格式不对的参数——字段名错了、类型不匹配。这需要在执行层做参数校验，校验失败时把错误信息返回给 LLM，让它重新生成。

**工具超时或无响应**
如果不做超时兜底，Loop 会在这一步卡死。成熟的 Agent 系统会给每个工具设置超时时间，超时后返回明确的错误信息，而不是一直等待。

工具的健壮性设计，是 Agent 工程里很重要的一环。一个脆弱的工具层，会让上层再好的推理逻辑也失效。


## 五、Tool Calling 和函数调用有什么区别

对有编程背景的读者，这里有一个值得注意的认知区别。

传统函数调用的路径通常由代码显式控制：什么时候调用、调用哪个函数、参数从哪里来，主要由程序逻辑决定。

Tool Calling 是**由 LLM 动态决定**的：调用时机、选哪个工具、参数填什么，都来自模型推理，不是硬编码。同样的任务，不同时刻可能走不同的工具路径。

这个区别带来了灵活性——Agent 能应对各种非预期的情况。但也带来了不确定性——你无法保证 LLM 每次都做出最优决策。

怎么衡量 Agent 的工具调用质量？怎么发现它在哪里出了问题？这是 Eval 要解决的事，后续文章会专门拆。


## 最后说一句

Tool Calling 是 Agent 伸向外部世界的手。

没有它，LLM 只是一个文字游戏机。有了它，Agent 才能真正采取行动、获取信息、影响系统。

但这只手怎么注册、怎么管理权限、怎么保证安全——这些是 Agent Runtime 负责的事。
