# Agent 怎么"记住"东西？一文讲透 Memory 体系

你有没有遇到过这种情况：

同一个问题问了 Agent 两次，第一次它给出了很好的分析。第二次问，它从头开始，完全不记得上次做过这件事。

不只是跨 session。有时候同一个 session 里，Loop 跑了二十几轮，它也开始"忘掉"前面发现的线索，绕回已经排查过的方向。

这通常不是单纯的 Bug，而是 Agent 记忆设计和上下文管理共同决定的结果。


## 先说清楚：这篇和 Context Engineering 那篇有什么不同

上一篇 Context Engineering 讲的是：每次调用 LLM 时，怎么管理塞进窗口的信息。

这篇讲的是更大的问题：**Agent 的记忆从哪里来，存在哪里，怎么跨轮次、跨 session 保留有用的信息。**

Context Engineering 是执行策略，Memory 体系是架构设计。两者有交叉，但视角不同。


## 一、为什么 Agent 需要显式设计记忆

两个约束决定了 Agent 的记忆问题不会自动解决。

**约束一：LLM 本身无状态。**
每次调用 LLM，它只能看到这次请求里的内容。没有"上次我们聊过什么"，没有"我之前做过这件事"。

**约束二：Context Window 有限。**
Session 内的信息最终要通过 Context 被模型看到，但不一定全都放在窗口里；很多系统会把原始状态存在外部，需要时再摘要或检索进来。窗口大小有上限，放进去的内容一旦超出，早期的内容就会被挤出。

这两个约束叠加，意味着：**如果不显式设计记忆系统，Agent 就是一个有选择性失忆的系统。**


## 二、理解 Agent 记忆的四个层次

不同产品对记忆的实现方式差异很大，但有一个分析框架可以帮助理解：把 Agent 的记忆分成四个层次，每层解决不同的问题。

> 📌 【插图一：四层记忆结构图】

**第一层：In-Context Memory（上下文记忆）**

就是当前 Context Window 里的所有内容——对话历史、工具返回、当前任务状态。

最直接，LLM 可以直接推理。对模型当前这次调用来说最即时；但如果系统不额外保存和重新注入，session 结束后它就不会自动延续。这是 Context Engineering 重点处理的部分。

**第二层：Working Memory（工作记忆）**

当前任务的精炼状态摘要：已知什么、排查了什么、下一步方向是什么。

不是把所有历史都保留，而是每轮结束后提炼关键信息，保持精简。专门为长 Loop 设计——当 In-Context Memory 开始膨胀，Working Memory 是它的压缩版。

**第三层：External Memory（外部存储）**

超出 Context 范围的信息，存到外部系统——文件、数据库、向量库。需要时按需检索，不一次性塞进 Context。

适合：大量历史数据、知识库、跨 session 需要保留的信息。RAG 就是 External Memory 的一种典型用法：把外部资料按需检索，再注入当前 Context。

**第四层：Episodic Memory（情节记忆）**

跨对话、跨任务的经验积累。"这个服务历史上出现过三次同类故障"、"上次这个问题最终根因是数据库连接池耗尽"。

帮助 Agent 从历史中学习，而不是每次从零开始。这是最高级、也是工业界目前实现最不成熟的一层，多数产品还在探索中。

这不是唯一的分类方式，也不是所有产品都会按这四层命名；它更像一个分析框架，帮助我们看清不同记忆机制各自解决的是哪类问题。


## 三、真实产品怎么做的

框架是框架，看看真实产品怎么实现，会更有感知。

以 Claude Code 为例，官方明确支持两类记忆：

**CLAUDE.md**：人工维护的持久指令。项目约定、技术栈说明、你不想让它重复问的问题——写在这里，每次 session 开始自动加载。更像持久化的项目指令，启动时直接注入上下文。

**Auto Memory**：模型自动维护的经验笔记。Claude Code 在工作过程中自己判断哪些信息值得记录，写入 `~/.claude/projects/<project>/memory/` 目录下的 Markdown 文件。

实现上刻意保持简单——没有向量库，没有语义检索，就是普通文件。`MEMORY.md` 作为索引，每次 session 开始自动加载前 200 行；其他主题文件（如 `debugging.md`、`api-conventions.md`）不预加载，Claude 需要时按需读取。

所有记忆都是普通 Markdown，完全透明可审计——随时可以打开修改，也可以在 session 里输入 `/memory` 查看当前加载了什么。

除此之外，长期运行的程序化 Agent 还常会设计**按需读写的 Memory Tool**——Agent 主动决定把什么存进去、什么时候取出来，而不是全量加载。这更接近 External Memory 的工程形态。

不同 Coding Agent 的跨 session 记忆能力差异很大：有的依赖显式项目文件，有的依赖自动摘要，有的则需要第三方工具或自研存储来补齐。

## 四、四层记忆怎么配合

四层记忆不是互斥的，是互补的。用一次故障排查任务来说明各层怎么分工：

任务开始，用户描述故障现象，这进入 **In-Context Memory**。

Agent 跑了五轮，查了日志、分析了调用链、排除了两个假设。这些进展提炼成摘要，写入 **Working Memory**，原始的工具返回不全量保留。

排查过程中需要查这个服务三个月内的历史告警，从 **External Memory** 检索相关记录注入 Context。

任务结束，根因是数据库连接池耗尽。这条经验写入 **Episodic Memory**，下次同类故障，Agent 第一轮就能把这个方向列为重点假设。

## 五、设计记忆体系时要做的几个决策

如果你要自己设计 Agent 的记忆体系，有几个问题需要想清楚：

**什么信息该进 Working Memory，什么该存 External Memory？**
Working Memory 存的是"当前任务进展"，随任务结束而归档；External Memory 存的是"跨任务可复用的知识"。关键区别是：这条信息对未来的任务有没有价值？

**External Memory 用向量库还是关系型数据库？**
向量库适合语义检索——"找和这次故障相似的历史案例"；关系型数据库适合精确查询——"查这个服务在某个时间段的所有告警"。很多成熟系统会两者结合。

**记忆怎么老化和清理？**
记忆不是越多越好。过时的信息会干扰当前推理。需要设计记忆的生命周期：什么时候更新，什么时候标记为过时，什么时候清除。


## 最后说一句

Agent 的记忆不是一个 Context，而是分层设计的系统——各层各司其职，互相配合。

设计得好，Agent 能从经验中积累，越用越顺手。设计得差，Agent 就是一个每次都从零开始的工具。
