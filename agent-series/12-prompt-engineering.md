# Agent 的行为说明书：一文讲透 Prompt Engineering

同一个 Agent，同一套工具，同一个任务。

换一版 System Prompt，结果可能截然不同。

第一版：Agent 查完日志，直接给出结论，言简意赅。
第二版：Agent 查了同样的日志，开始东拉西扯，扯到了不相关的系统，最后结论模糊。
第三版：Agent 拒绝回答，说"这超出了我的能力范围"。

三个版本，工具没变，模型没变，唯一变的是 System Prompt。

Prompt 不只是"给模型的一段话"。它是 Agent 的行为说明书——影响 Agent 如何理解角色、如何使用工具、如何输出结果，以及遇到边界情况时如何处理。

---

## 一、为什么 Prompt Engineering 对 Agent 尤其重要

对于简单的问答场景，Prompt 写得粗糙一点，影响不大——模型理解问题，给出答案，用户满意。

Agent 不一样。Agent 要执行多步骤任务，要调用工具，要做决策，要和外部系统交互。每一步决策都受 System Prompt 影响。一个模糊的 Prompt，会在每一轮循环里积累偏差，到最后放大成完全跑偏的行为。

具体来说，System Prompt 对 Agent 的影响体现在四个维度：

**角色与定位**：Agent 是谁、有什么专业背景、面对什么用户——这决定了它的"默认行为模式"。一个被定义为"严谨的故障排查专家"的 Agent，和一个没有角色定义的 Agent，在面对模糊问题时的处理方式完全不同。

**工具使用行为**：什么情况下该调工具、调哪个工具、调之前要不要先推理——这些行为如果不在 Prompt 里显式约束，模型会按自己的"直觉"决定，结果难以预期。

**输出格式**：结构化输出还是自由文本、用什么字段、每个字段的含义——如果下游系统要解析 Agent 的输出，格式不稳定是灾难性的。

**边界与安全**：哪些事不能做、遇到不确定情况怎么处理、什么时候该说"我不知道"——这是 Agent 行为的底线，必须在 Prompt 里明确。但高风险边界还要由 Runtime、权限系统和 Guardrail 强制执行，Prompt 本身是软约束。

## 二、System Prompt 的基本结构

一个工程化的 Agent System Prompt，通常包含以下几个部分：

**角色定义**

告诉模型它是谁，有什么专业能力，服务于什么场景。

```
你是一个线上故障排查专家，专注于分布式系统的性能问题诊断。
你的用户是有后端开发经验的工程师，不需要解释基础概念。
```

**任务说明**

说清楚 Agent 的核心任务是什么，以及完成任务的基本流程。

```
你的任务是：根据用户描述的故障现象，通过调用工具收集证据，
给出根因分析和处理建议。

基本流程：
1. 理解故障现象
2. 制定排查计划
3. 调用工具收集证据
4. 分析根因
5. 给出可操作的建议
```

**工具使用规范**

告诉模型什么时候用工具、怎么用、用完之后做什么。

```
工具调用规范：
- 每次工具调用前，先说明你为什么要调用这个工具
- 拿到工具结果后，先分析结果再决定下一步
- 不要在没有证据的情况下给出根因结论
- 如果工具返回空结果，说明原因并尝试其他查询方式
```

**输出格式**

约定最终输出的结构，并尽量配合 structured output 或 schema validation，让下游系统可以稳定解析。

```
最终输出格式：
{
  "root_cause": "根因描述",
  "evidence": ["证据1", "证据2"],
  "confidence": "high/medium/low",
  "recommendations": ["建议1", "建议2"]
}
```

**边界约束**

明确不能做什么、遇到不确定情况怎么处理。

```
边界约束：
- 不要直接执行任何写操作，写操作必须等待人工确认
- 如果证据不足以得出结论，明确说明"证据不足，需要进一步收集"
- 不要猜测根因，所有结论必须有工具返回的数据支撑
```

需要特别注意：这类规则不能只靠 Prompt，工具执行层也必须做权限校验和 HITL 拦截。Prompt 里的约束是行为引导，不是安全保障。

**Prompt 是软约束，不是安全边界**

需要特别强调：Prompt 是行为引导，不是安全机制。它能告诉模型应该怎么做，但不能保证模型一定照做。凡是涉及权限、写操作、敏感数据和合规要求的边界，都必须在 Runtime、工具层或 Guardrail 层做硬校验。把安全责任全部压在 Prompt 上，是生产系统最常见的设计误区之一。

## 三、几个容易踩的坑

**坑一：指令太模糊**

"尽可能帮助用户解决问题"——这句话对模型来说几乎没有约束力。遇到边界情况，模型会自己决定怎么做，结果难以预期。

好的 Prompt 要具体：什么情况下做什么，给模型明确的行为路径，而不是让它自由发挥。

**坑二：指令互相矛盾**

"你要全面分析所有可能的原因"和"你的回答要简洁，不超过 200 字"——这两条放在一起，模型不知道该听哪个。

写 Prompt 时要检查指令之间是否有冲突，矛盾的指令会让模型行为变得随机。

**坑三：没有处理不确定情况的指令**

当证据不足、工具失败、用户描述模糊时，模型该怎么做？如果 Prompt 里没有说，模型可能选择继续猜测，给出一个看似有理但实际无据的结论。

要显式告诉模型：不确定的时候，说不确定。

**坑四：格式约定不够严格**

"以 JSON 格式输出"——但没有说具体有哪些字段、字段的类型是什么、缺失值怎么处理。结果是：模型有时输出 `root_cause`，有时输出 `rootCause`，有时根本没有这个字段。

最好使用 API 提供的 structured output / JSON schema 来约束输出格式；如果只能靠 Prompt，也要给出字段名、字段类型、缺失值处理方式和一个完整的输出示例。

## 四、动态 Prompt：运行时注入内容

静态 System Prompt 解决的是"默认行为"的问题。但 Agent 在运行时还需要注入动态内容：

**当前任务上下文**：这次任务的具体信息——用户是谁、涉及哪个系统、已知的背景信息。

**工具列表**：当前可用工具及其描述。有些系统会通过 tools schema 直接传给模型（现代 tool calling API 的标准方式），有些会作为提示词内容注入。两种方式各有适用场景，不同场景下 Agent 能用的工具不同，动态管理比静态写死更灵活。

**历史状态**：已经收集了哪些证据、已经排除了哪些可能性——这些中间状态注入到 Prompt 里，帮助模型做连贯的推理。

**约束条件**：这次任务特有的限制——比如"只能查只读接口"、"时间窗口限制在最近 1 小时"。

动态注入的内容和静态 System Prompt 共同构成模型每一轮推理的完整输入。这正是 Context Engineering 的核心——不只是写好 System Prompt，而是在每一轮都精准控制模型能看到什么。

## 五、Prompt 也需要版本管理和测试

Prompt 改动的影响范围不亚于代码改动。改了一句话，Agent 的行为可能在某些边界情况下发生意想不到的变化。

工程上对待 Prompt 应该和对待代码一样：

**版本管理**：每次改动都记录，能回溯。改坏了能回滚。

**回归测试**：每次改 Prompt 之后，跑一遍 Eval，确认核心 case 的表现没有退步。这正是 Eval 体系的价值所在——Prompt 迭代和 Eval 回归是配套的。

**灰度发布**：重大改动不要一次全量上线，先在部分流量上验证效果。

Prompt 不是一次写好就不用管的东西。随着任务场景扩展、边界 case 积累、工具集变化，Prompt 需要持续迭代。把它当成代码来维护，而不是当成配置来随意修改。

---

## 六、没有好 Prompt 的代价

一个 System Prompt 写得粗糙的 Agent：

**行为不可预期**：同样的输入，不同时刻给出不同的行为，你不知道是因为模型随机性还是 Prompt 有歧义。

**边界失控**：没有明确约束的 Agent，会在你不期望的地方做出你不期望的决定——调用了不该调的工具，给出了没有根据的结论，在该停下来问的时候径直往前冲。

**格式不稳定**：下游系统解析输出时频繁报错，因为 Agent 每次输出的结构都不一样。

有了工程化的 System Prompt，Agent 的行为才有了可预期的约束。它不能保证 Agent 永远正确，但能显著降低行为漂移的概率；真正的硬边界还要靠 Runtime、Eval、HITL 和 Guardrail。

---

## 最后说一句

Prompt 是 Agent 和模型之间的契约。

你在 Prompt 里写清楚的，模型会尽力遵守；你没写清楚的，模型会自己决定——而那个决定，可能不是你想要的。

写好 System Prompt，不是为了"调教"模型，而是为了让 Agent 的行为可预期、可测试、可维护。这和写好接口文档、写好单测，是同一件事。
