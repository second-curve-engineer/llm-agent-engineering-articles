# Context Engineering：Agent 真正的难点在这里

上一篇讲 Agent Loop 的结尾，留了一个问题：

> Loop 本身不复杂，`while True` 一个下午能写完。真正难的是每次调用 LLM 时，怎么把"已经发生的事"塞进去。

这篇就来回答这个问题。

---

## 先从一个反直觉的现象说起

同样一个故障排查 Agent，处理同一类报错：

第一次，它三轮之内定位了根因，分析准确。

第二次，它兜兜转转跑了十几轮，最后给出一个驴唇不对马嘴的结论。

代码没变，工具没变，任务也差不多。唯一的区别是：两次塞给 LLM 的信息不一样。

这背后有一个很多人没意识到的事实：**LLM 本身是无状态的。每次调用时，它只能看到这次请求里提供的内容；窗口之外的历史，除非系统重新塞回来，否则它并不知道。**

这个窗口，就是 Context Window。

---

## 一、Context Window 是什么

打一个比方。

你去找一位顾问咨询，但这位顾问有个奇怪的毛病：每次见面都会失忆，完全不记得上次谈了什么。

所以每次见面前，你必须把所有相关背景写在一张纸上递给他。他只根据这张纸回答你，纸上没写的事，他一概不知。

这张纸，就是 Context Window。

在 Agent 里，每次调用 LLM，这张纸上通常写着四类内容：

- **System Prompt**：告诉 LLM 它是谁、能用哪些工具、遵守哪些规则
- **执行历史**：之前每轮的模型回复、工具调用与返回结果
- **工具返回结果**：上一步调用工具拿到的数据
- **用户输入**：最初的任务描述

> 📌 【插图一：Context Window 构成示意图】

这张纸的大小是有上限的。如果内容超出上限，轻则被系统截断或压缩，重则 API 直接报错。无论哪种情况，模型最终都只能基于窗口里的内容工作。

---

## 二、Agent 跑起来之后，Context 会发生什么

单次对话还好。一旦 Agent 开始跑 Loop，问题就来了。

还是用故障排查的例子。

第 1 轮：查日志，拿到一段错误输出。
第 2 轮：读代码，拿到相关文件内容。
第 3 轮：搜调用链，拿到一串调用记录。
……

最朴素的做法，是每一轮都把工具返回结果追加进 Context。但 Agent 一跑久，这种做法很快会出问题。到第 10 轮，Context 里已经堆满了：原始日志、代码片段、调用记录、每轮的回复……

这就是 **Context 爆炸**。

> 📌 【插图二：Context 随 Loop 轮数膨胀的过程图】

Context 爆炸会带来三种问题：

**1. 直接超限报错**
最显性。内容超出窗口上限，API 直接返回错误，Agent 崩掉。

**2. 早期信息被截断**
没有超限，但为了塞进新内容，早期的信息被挤出了窗口。Agent "忘了"第 1 轮发现的关键线索，开始原地打转。

**3. 注意力被稀释**
这是最隐蔽的问题。窗口没超限，但里面堆满了冗余信息——重复的日志、无关的代码片段、早已过时的推理过程。LLM 淹没在噪音里，找不到重点，回答质量悄悄变差。

前两种问题容易发现，第三种往往让人误以为是"模型能力不够"，其实是 Context 质量太差。

---

## 三、什么是 Context Engineering

到这里，问题已经很清楚了：**Context 不是越多越好，而是越精准越好。**

Context Engineering，就是在有限的窗口里，**决定放什么、不放什么、怎么放**。

常见的几种策略：

**压缩历史**
不把每一轮的完整过程都保留，而是定期把前几轮总结成一段摘要，只保留关键结论。就像开会记录不会逐字逐句，只记重点。

**选择性保留**
不是所有工具返回都值得进 Context。一段 500 行的日志，真正有用的可能只有 3 行报错。只保留和当前任务直接相关的部分。

**Working Memory**
维护一个持续更新的"任务状态摘要"——当前已知什么、已经排查了什么、下一步方向是什么。每轮结束后更新这份摘要，而不是把原始过程全部保留。

**RAG（检索增强）**
把大量背景知识存在外部，不一次性塞进 Context，而是根据当前任务需要，按需检索最相关的片段放进去。

所以 Context Engineering 不只是"把内容变短"，而是在设计 Agent 的记忆系统：哪些信息进入短期上下文，哪些沉淀成 Working Memory，哪些放到外部存储，哪些干脆丢弃。

---

## 四、一个对比

还是故障排查的场景，同一个 Agent，两种策略：

**策略 A：堆料**
每轮结束，把完整的工具返回、完整的推理过程全部追加进 Context。跑到第 8 轮，窗口里装满了原始日志、重复的错误信息、已经被推翻的假设……LLM 在噪音里迷失，开始给出模糊的结论。

**策略 B：精炼**
每轮结束，只把"这轮发现了什么"提炼成一句话写进 Working Memory，原始数据不直接塞进 Context，而是保存在外部，需要复核时再取回。第 8 轮时，Context 里是一份干净的排查进展摘要，LLM 清楚地知道目前掌握了哪些线索，下一步该往哪走。

同样的任务，同样的模型，结果可能天壤之别。

## 最后说一句

有一个说法流传很广：**Prompt Engineering 教你怎么问问题，Context Engineering 教你怎么管信息。**

在单次对话里，Prompt 写得好不好很重要。但在 Agent 里，Loop 一旦跑起来，真正决定输出质量的是 Context 的质量——你塞给 LLM 的是精准的信息，还是一堆噪音。

这是 Agent 工程里最容易被忽视、也最值得投入的部分。这篇讲的是最基础的 Agent 模型；工业级 Agent 在 Context 管理上会复杂得多——外部存储、分级记忆、动态裁剪……这些是后续文章会展开的方向。
