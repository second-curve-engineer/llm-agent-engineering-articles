# Agent 怎么想清楚再动手：一文讲透 Planning

给 Agent 一个任务："帮我排查这次线上故障，给出根因和处理建议。"

Agent 开始跑了。

第一步，它查了一下最近的报警记录。
第二步，它去看了数据库慢查询日志。
第三步，它突然去查了一下服务器的 CPU 使用率。
第四步，它又回来看了一遍报警记录。
第五步，它调用了一个和这次故障完全不相关的工具。

十几轮之后，它给出了一个结论——但你不确定它是怎么得出来的，也不知道它有没有遗漏关键信息。

这就是缺少显式 Planning 的 Agent：每一步都能动，但整体路径不够可控。



## 一、为什么 Agent 需要 Planning

对于简单任务，Agent 通常不需要单独设计显式 Planning。"查一下这个服务的错误率"——直接调工具，一步出结果，没有规划的必要。

但复杂任务不一样。复杂任务有三个特征，让"直接动手"变得危险：

**目标和行动之间有距离**：用户说的是"排查故障"，不是"调用哪个工具"。Agent 需要自己把目标分解成可执行的步骤，这个分解本身就需要推理。

**步骤之间有依赖**：必须先确认报警类型，才能知道该看哪个日志；看完日志，才能判断是不是数据库问题。顺序错了，后面的步骤都是在浪费工具调用。

**资源有限**：Context 窗口有限，工具调用有成本，时间有限制。随机探索的代价太高，需要一条更高效的路径。

Planning 解决的核心问题是：**在动手之前，先想清楚怎么做。**



## 二、Planning 是什么

Planning 就是 Agent 在执行任务之前（或执行过程中），对目标进行分解、排序、调整的能力。

一个有 Planning 能力的 Agent，面对复杂任务时会做三件事：

**目标分解**：把"排查故障"拆成"确认报警类型 → 收集相关日志 → 分析根因 → 给出建议"。

**步骤排序**：识别步骤之间的依赖关系，确定哪些必须串行、哪些可以并行。

**动态调整**：执行过程中发现新信息，能修正原来的计划——比如查日志发现不是数据库问题，及时转向查网络层。

## 三、几种主要的 Planning 范式

Planning 不是一种固定的实现方式，而是一类能力。工程上有几种常见的范式，本质区别在于：**什么时候想，想多深。**

**ReAct：边想边做**

ReAct（Reasoning + Acting）不是严格的全局规划方法，更像一种边推理边行动的执行范式，但它承担了最基础的局部规划作用。Agent 在每一轮循环里先输出一段推理（Thought），再决定下一步行动（Action），看到结果（Observation）之后继续推理。

```
Thought: 报警显示是数据库超时，应该先查慢查询日志
Action: query_slow_log(last_30min)
Observation: 发现 3 条超过 5s 的查询，都来自同一张表
Thought: 需要进一步确认这张表的索引情况
Action: check_table_index(table_name)
...
```

优点是灵活，能根据每一步的结果动态调整。缺点是"走一步看一步"，对于需要全局规划的复杂任务，容易走弯路。

**Plan-and-Execute：先想后做**

先由 Planner 生成一份初始执行计划，再交给 Executor 按计划执行，并在必要时触发重新规划。两个角色分离：Planner 负责全局规划，Executor 负责逐步执行。

```
Planner 输出：
1. 查询最近 30 分钟的报警记录
2. 根据报警类型，查对应的慢查询日志
3. 分析慢查询的根因
4. 检查相关表的索引状态
5. 汇总生成报告

Executor: 按顺序执行上述步骤
```

优点是执行路径清晰，减少随机漫步。缺点是计划一旦生成就相对固定，遇到意外情况需要重新规划，灵活性不如 ReAct。

**RePlan：计划 + 动态修正**

ReAct 和 Plan-and-Execute 的折中。先生成一个初始计划，执行过程中如果发现计划需要调整，重新规划剩余步骤。既有全局视角，又保留了动态调整的能力。

适合任务目标清晰但执行过程中不确定性较高的场景。

**ToT（Tree of Thoughts）：多路探索**

对于需要深度推理的任务，Agent 同时探索多条思路，评估每条路径的可行性，选择最优路径继续。类似人类做难题时的"先想几种方案，再选最好的"。

在某些需要多路径推理和方案比较的任务上，ToT 可能提升效果，但代价是更高的 Token 消耗、更长的延迟和更高的实现复杂度。实际工程中通常只在关键决策节点使用，而不是全程开启，也常被其他更轻量的策略替代。

**Planning 的风险**

Planning 不是把计划写得越长越好。计划太细，会让 Agent 被早期假设绑死——任务刚开始时信息不完整，过度细化的计划反而成了枷锁；计划太粗，又起不到约束作用，和没有 Planning 没有本质区别。

好的 Planning 应该是可执行、可检查、可调整的：每一步有明确目标，执行后能更新状态，必要时能重新规划。这才是 Planning 真正的工程价值所在。


## 四、工业界怎么实现

**LangGraph**

天然支持 Plan-and-Execute 和 ReAct 两种模式。通过图结构定义执行流程，Planner 节点负责生成计划，Executor 节点负责执行，支持在执行过程中动态修改图的状态。官方有现成的 Plan-and-Execute 示例。

**LlamaIndex Workflows**

以事件驱动的方式组织 Agent 执行流程。每个步骤是一个 Step，步骤之间通过事件触发。支持把复杂任务拆成多个子步骤，天然适合 Plan-and-Execute 模式。

**OpenAI Agents SDK**

可以通过多个 Agent 的职责划分、结构化输出和 Handoff 实现 Planner / Executor 分离：Planner Agent 生成结构化计划，Runtime 校验后通过 Handoff 把控制权交给对应的 Executor Agent 或工具执行。每个 Agent 职责单一，边界清晰。

**自研 Planner**

很多生产系统会自己实现 Planner 层——用一个专门的 LLM 调用生成结构化的执行计划（JSON 格式），再交给 Runtime 按计划调度工具。这样可以对计划做校验、做权限检查，在执行之前就发现明显错误。

生产系统里通常还会维护 plan state：哪些步骤已完成、哪些被跳过、哪些需要重试，以及每一步对应的证据。这是 Planning 工程化的关键——计划不只是一份清单，而是一个有状态的执行上下文。



## 五、没有 Planning 的代价

没有 Planning 的 Agent，在简单任务上表现还好，一碰到复杂任务就暴露问题：

**路径混乱，效率低**：每一步靠 LLM 临时决定，没有全局视角，容易绕远路、重复调用、遗漏关键步骤。

**难以调试**：执行路径每次不一样，出了问题不知道问题出在决策上还是执行上。

**成本失控**：没有计划的约束，Agent 可能调用大量不必要的工具，Token 消耗和 API 费用都难以预期。

有了 Planning，Agent 才从"随机漫步"变成"有目的的行动"。执行路径可预期，中间过程可检查，出了问题可以定位到具体哪一步的计划出了偏差。



## 最后说一句

Planning 是 Agent 从"能完成简单任务"到"能处理复杂问题"的关键跨越。

一个没有 Planning 的 Agent，就像一个拿到任务就埋头干、从不想清楚再动手的工程师——短任务还好，长任务必然翻车。

但 Planning 也不是越复杂越好。ReAct 的灵活、Plan-and-Execute 的清晰、ToT 的深度——选哪种，取决于任务的复杂度和对可控性的要求。
