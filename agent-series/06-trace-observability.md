# Agent 的黑匣子：一文讲透 Trace 与可观测性

Agent 跑了 10 轮，给出了一个错误的结论。

你想排查问题——是第 3 轮工具返回有误？还是第 7 轮 LLM 推理跑偏了？还是某个工具参数填错了？

打开日志，看到的是一堆零散的输出，完全串不起来。你不知道从哪里下手。

这就是没有 Trace 的 Agent：能跑，但出了问题是个黑盒。


## 一、为什么 Agent 特别需要可观测性

传统后端服务出了问题，通常能靠日志和错误码定位——接口返回了 500，去查对应的异常堆栈，链路清晰。

Agent 不一样，有三个特殊的地方：

**推理不确定**：LLM 的输出不是确定性的。同样的输入，不同时刻可能走不同的推理路径。出了问题，你很难复现，也很难判断是偶发还是必现。

**多步骤因果链**：一个任务可能跑 10 轮以上，每一轮的输出都影响下一轮的决策。第 3 轮工具返回了错误数据，可能到第 8 轮才表现出明显的异常——用传统日志根本看不出因果关系。

**工具有副作用**：Agent 会真实地调用外部系统——查数据库、发请求、修改文件。出了问题不只是"答案不对"，可能是"真的做了不该做的事"，事后复盘更加重要。

这三点加在一起，决定了 Agent 必须有专门设计的可观测性体系。


## 二、Trace 是什么

Trace 就是把 Agent 每一轮 Loop 的完整执行过程记录下来。

每一轮记录的内容包括：

- 这一轮的输入（Context 里有什么）
- LLM 输出了什么（最终回复、工具调用请求、必要的中间状态）
- 工具名称、传入参数
- 工具返回了什么
- 这一轮耗时多少、用了多少 Token
- 有没有报错

把所有轮次串起来，加上一个唯一的 `run_id`，就是一条完整的执行链路——从用户输入到最终结论，中间每一步都有迹可查。

需要注意的是，Trace 不是把所有原始内容无脑存下来。上下文、工具结果、用户数据里可能包含敏感信息，工程上通常要做采样、脱敏、权限控制和保留周期管理。记录的目标是保留可复盘的关键证据，而不是完整镜像每一个字节。

**和传统后端 Trace 有什么不同？**

有后端经验的读者可能熟悉分布式链路追踪——一个请求经过多个服务，每个服务产生一个 Span，串成一条 Trace。

Agent Trace 借用了同样的概念，但追踪的内容完全不同：传统 Trace 追的是"服务调用了什么"，Agent Trace 追的是模型看到了什么、输出了什么、调用了什么工具、工具返回了什么，以及这些步骤之间的因果关系。这些是传统 APM 工具完全覆盖不了的。

## 三、Observability 是什么

Trace 是原始记录，Observability 是在这些记录上建立的能力。

有了 Trace 数据，可以做三件事：

**实时监控**：Agent 现在跑到第几轮？有没有卡住？工具调用成功率怎样？Token 消耗速度是否正常？这些指标能让你在任务还在跑的时候就感知到异常。

**异常检测**：哪一轮 Token 消耗异常高？哪个工具频繁失败？哪类任务的完成率特别低？这些是系统性问题的信号，不靠 Trace 数据根本发现不了。

**复盘分析**：这次任务为什么失败？从哪一轮开始跑偏的？LLM 在哪一步做了错误的决策？有了完整的 Trace，复盘才能精确到具体的轮次和具体的工具调用。

Trace 解决"记录了什么"，Observability 解决"能看到什么、能发现什么"。

## 四、工业界怎么做

工程上，很多 Agent Observability 系统会复用 OpenTelemetry 的 Trace / Span 模型，并在此基础上扩展 GenAI 相关字段。常见做法是用一个 root trace 表示一次 Agent run，再把每轮 Loop、LLM 调用、工具调用、检索、handoff 等记录成不同层级的 Span。

目前工业界有几个主流的 Agent Observability 平台：

**LangSmith**：LangChain 官方出品，对 LangGraph 支持最深，适合 LangChain 技术栈。

**Langfuse**：开源、可自部署，2026 年初被 ClickHouse 收购，是很多团队做自建 LLM Observability 时会重点考虑的方案。

**Arize Phoenix**：有较强的 ML Observability 和 Eval 背景，适合希望把 tracing、dataset、experiment、eval 串起来的团队。

这些平台虽然生态和侧重点不同，但都在解决同一个核心问题：捕获完整 session 或 run 的执行链路，而不只是记录单次 LLM 调用。这个差别很关键——Agent 的问题往往不在某一次调用，而在多步骤的因果链上。


## 五、没有 Trace 的代价

没有 Trace 的 Agent，会面临三个实际问题：

**出了问题不知道为什么**：只能靠猜，改了一个地方不确定是不是真的修好了。

**优化没有依据**：想提升 Agent 的准确率，但不知道瓶颈在哪——是工具质量问题、是 Prompt 问题、还是 Context 管理问题？

**上线不敢放心用**：Agent 做了什么你不知道，用户反馈了问题你查不清楚，对系统的信任建立不起来。

有了 Trace，Agent 才从"黑盒工具"变成"可信系统"。这不是锦上添花，是工程上能不能用的基础门槛。


## 最后说一句

Trace 是 Agent 的黑匣子，Observability 是飞行员的仪表盘。

两者加在一起，让你知道 Agent 在做什么、做对了没有、出了问题去哪找。Agent 从"能跑"到"敢用"，靠的就是这个。
