# 给 Agent 打分：一文讲透 Eval

Agent 跑完了，给出了一份故障排查报告。

结论看起来挺像那么回事：定位到了慢查询，列出了三个可能根因，给出了处理建议。

但你怎么知道它是真的对了？还是只是"看起来对"？

你改了一版 Prompt，重新跑了一遍，结论稍微不一样了——是变好了还是变差了？

没有 Eval，你只能靠感觉。靠感觉的系统，不叫工程，叫碰运气。



## 一、为什么 Agent 特别需要 Eval

传统软件很多场景可以靠单测、集成测试来验证：给定输入，检查输出是否符合预期。

Agent 不一样，有三个特殊的地方：

**输出不确定**：同样的输入，LLM 在不同时刻可能给出不同的推理路径和结论。你没法用一个固定的 `assertEqual` 来判断对错，需要的是评判标准，而不只是断言。

**多步骤放大误差**：Agent 跑 10 轮，早期第 2 轮的一个小偏差，可能在第 8 轮被放大成完全跑偏的结论。单看最终输出，你不知道问题出在哪一步。

**没有 Eval 就没有迭代依据**：换了模型、改了 Prompt、调整了工具——你不知道这些改动是让 Agent 变好了还是变差了。没有量化指标，优化就是瞎改。

这三点加在一起，决定了 Eval 不是锦上添花，而是 Agent 能不能持续迭代的基础。



## 二、Eval 是什么

Eval 就是用一套可复现的标准，衡量 Agent 在任务结果、执行过程、成本和安全边界上的表现。

注意，不只是评"答案对不对"。Agent 的质量是多维的：

**三个层次**

**单步 Eval**：评判某一次 LLM 调用或工具调用的质量。它是否该调用工具？工具选对了吗？参数 schema 是否正确？工具结果是否被正确吸收？单步 Eval 粒度最细，适合定位具体哪个环节出了问题。

**轨迹 Eval**：评判整条执行链路的合理性。Agent 有没有走对路径？工具调用顺序合不合理？有没有绕远路、兜圈子、重复调用同一个工具？轨迹 Eval 还会关注成本和风险：是否用了过多轮次，是否调用了不必要的高风险工具，是否在该触发 HITL 时没有暂停。

**端到端 Eval**：评判最终结论的质量。给定用户的问题，Agent 的最终答案对不对、完不完整、有没有遗漏关键信息？这是用户最直接感受到的质量。

> 📌 【插图一：Eval 三个层次示意图】

三个层次要配合使用。只看端到端，出了问题不知道根因在哪；只看单步，可能每步都"对"但整体跑偏了；轨迹 Eval 是连接两者的桥梁。

**常见的评测指标**

所以 Agent Eval 不是只问"答案对不对"，还要看：任务成功率、工具调用正确率、平均轮次、Token 成本、延迟、是否触发了必要的 HITL、是否违反权限或安全策略。这些指标加在一起，才能反映一个 Agent 系统的整体质量水位。



## 三、怎么评

知道评什么之后，还要知道怎么评。常见的方式有四种，各有适用场景：

**基于规则**：格式检查、关键字匹配、结构校验。比如"输出必须包含 root_cause 字段"、"工具调用次数不超过 5 次"。实现简单、运行快、结果稳定，但覆盖范围有限，只能判断"有没有"，判断不了"好不好"。

**基于参考答案（Golden Dataset）**：准备一批有标注的测试用例，用标准答案、评分 rubric 或 checklist 来评估 Agent 输出。对故障排查这类任务，往往没有唯一正确答案，更适合用 rubric——根因是否覆盖、证据是否充分、建议是否可执行。难点在于维护数据集的成本，以及标准答案本身可能随时间过期。

**LLM-as-Judge**：用一个 LLM 来评判另一个 LLM 的输出。灵活、覆盖广，可以评判语义质量。但 Judge 本身有偏见，最好让它按明确 rubric 逐项评分，并要求引用输出中的证据，而不是只给一个总分——否则很容易打出玄学分。评分结果还需要和人工标注做校准，才能信任它的判断。

**人工标注**：成本最高，但是校准其他评估方式的基准。通常不用于全量评测，而是用于：建立初始 Golden Dataset、验证 LLM-as-Judge 的评分是否可信、处理边界 case。

实践中通常组合使用：规则做快速过滤，Golden Dataset 做回归，LLM-as-Judge 做质量评分，人工标注周期性校准。



## 四、数据集从哪里来

Eval 的质量上限取决于测试数据集的质量。数据集从三个地方来：

**从真实 Trace 里挖**：把线上跑过的 run 里，表现好的标注为 positive case，表现差的标注为 negative case，积累成 Golden Dataset。这是最贴近真实场景的数据来源，但需要做清洗、脱敏和标注，成本并不低。

**构造边界用例**：故意制造模糊场景——工具返回空结果、信息互相矛盾、用户描述不清晰。这些边界 case 在线上不一定常见，但最容易暴露 Agent 的脆弱性。

**持续扩充**：每次在线上发现新的 bad case，加进数据集，跑一遍回归。数据集不是一次性建完就放着，而是随 Agent 迭代一起生长的。

这三个来源加在一起，Eval 才能真正反映 Agent 在生产环境里的表现。



## 五、工业界怎么做

概念清楚之后，还需要知道用什么工具落地。目前 Agent Eval 工具分成商业平台和开源框架两类，各有定位。

**LangSmith**

LangChain 官方出品，和 LangChain / LangGraph 深度绑定。如果你的 Agent 是基于 LangGraph 搭建的，LangSmith 提供原生的 Trace、Eval 和 Prompt 管理，是这个技术栈里阻力最小的起点。跨框架使用需要额外适配。

**Braintrust**

定位是"把 LLM 质量当成发版门禁来管理"。数据集优先、模型无关，支持沙箱化的自定义 scorer，可以把 Eval 跑进 CI/CD 流程。适合已经建立了 Golden Dataset、希望每次发版前自动跑回归的团队。

**Langfuse**

开源可自部署，Trace 和 Eval 一体。适合对数据安全有要求、不想把数据发给第三方平台的团队。2026 年初被 ClickHouse 收购后继续开源，是自建 LLM Observability + Eval 方案里使用最广泛的选择之一。

**DeepEval**

开源框架，pytest 风格，可以直接跑在 CI 里。适合工程师在本地开发流程中做 LLM 行为验证，上手门槛低，适合还没有完整 Eval 基础设施的团队起步用。

这四个工具虽然形态不同，但解决的核心问题是一致的：**让 Eval 可复现、可追踪、可集成进迭代流程**——而不是每次靠人工跑几个例子凭感觉判断。



## 六、没有 Eval 的代价

没有 Eval 的 Agent，会卡在三个困境里出不来：

**优化靠感觉**：改了 Prompt，跑了几个例子，"感觉好多了"。但你不知道这几个例子是不是代表性的，也不知道改动有没有引入新的问题。

**上线没有信心**：不知道当前版本的质量水位在哪，不敢上线，或者上了线也提心吊胆。

**bad case 无法系统沉淀**：用户反馈的问题修了一个算一个，没有进入回归集，下次类似问题还会再犯。

有了 Eval，Agent 才有了可量化的质量基线。每次迭代都能看到分数变化，上线前能跑回归，出了问题能定位到具体哪一步。



## 最后说一句

Eval 是 Agent 的体检报告。

没有体检，你不知道它健不健康；有了体检，你才能有依据地迭代、有信心地上线。

Agent 从"能跑"到"跑得好"，靠的就是 Eval。
