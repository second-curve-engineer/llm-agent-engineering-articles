# 为什么 Agent 跑几轮之后开始"胡说"？

## Context Window 与 KV Cache 详解

---

你有没有遇到过这种情况：

Agent 刚开始跑得好好的，前几轮对答如流。但跑到第十几轮，它突然"忘了"你在第二轮设置的约束，开始重复已经做过的步骤，甚至给出和之前矛盾的结论。

你以为是 prompt 写得不好，或者工具调用出了问题。反复调试，没有结论。

其实问题的根源更底层——你没有理解 LLM 是怎么"看"这个世界的。

---

## LLM 没有记忆，只有一个窗口

人类在对话时，大脑会主动维护一份"上下文"：记住对方说过什么，结合前因后果来理解当前这句话。这个过程是持续的、有选择的，而且不受物理限制。

LLM 不是这样工作的。

每次你调用 LLM，它看到的不是"整个对话历史"，而是一段被拼接在一起的文本——从第一条系统提示，到所有的对话轮次，到当前这条用户输入。这整段文本，就是它的全部视野。

这个视野，就叫 **Context Window**（上下文窗口）。

窗口之外的内容，LLM 完全看不到，就像从未存在过一样。

这不是"记忆力差"，而是架构设计：LLM 是一个无状态的函数，输入是一段文本，输出是下一段文本。它没有跨调用的持久状态，每次推理都从零开始，唯一的信息来源就是这次输入的 context。

---

## 窗口有多大？

不同模型的 context window 大小不同，通常用 **Token 数**来衡量（Token 是什么我们下篇再讲，这里先理解为"大约等于汉字或单词的数量"）。

- GPT-3.5：4K tokens（早期版本）
- GPT-4o：128K tokens
- Claude 3.5 Sonnet：200K tokens
- Gemini 1.5 Pro：100万 tokens

128K tokens 大约是 10 万汉字，相当于一本中篇小说。听起来很多，但在 Agent 场景下，消耗速度极快：

- 每轮对话的用户输入和模型回复
- 工具调用的请求和返回结果
- RAG 检索出来的参考文档
- 系统提示（往往几百到几千 token）

一个复杂的 Agent 跑二三十轮，context 轻松突破 5 万 token。跑上百轮，128K 也不够用。

---

## 窗口满了会怎样？

这就是"胡说"的真正原因。

当 context 接近上限，工程上通常有两种处理方式：

**截断（Truncation）**：直接丢掉最早的内容，只保留最新的 N 个 token。副作用显而易见——系统提示、早期的关键约束、第一轮确认的目标，可能就这样被丢掉了。LLM 在"失忆"的状态下继续工作。

**报错或拒绝**：API 层面直接返回错误，整个 Agent 任务中断。

两种结果都很糟糕。而很多框架的默认行为是静默截断——你甚至不知道什么时候丢了内容，只看到模型开始答非所问。

这就是为什么理解 context window 对 Agent 工程师来说是基本功：你得知道你的 Agent 在什么时候开始"看不见"历史，并且提前设计好应对策略。

---

## KV Cache：让推理不那么慢

讲完 context window 的"边界问题"，再来看 KV Cache，它解决的是另一个问题：**推理速度**。

理解 KV Cache，需要先知道 Transformer 在做什么（详细原理在后面的 Attention 篇讲，这里只讲直觉）。

LLM 生成文本时，是一个 token 一个 token 地往外"吐"的。生成第 N 个 token 时，模型需要"回望"前面所有的 token，计算它们之间的关联关系——这个计算，就是 Attention。

问题来了：生成第 N 个 token 时，前面 N-1 个 token 的关联计算，在生成第 N-1 个 token 时**已经做过一遍了**。如果每次都重新算，随着序列变长，计算量会以平方级增长。

KV Cache 的思路很简单：**把算过的中间结果缓存起来，下次直接用**。

具体来说，Attention 计算过程中有两个关键的中间矩阵：K（Key）和 V（Value）。每生成一个新 token，只需要计算这个新 token 的 K/V，然后追加到缓存里，而不是把整个序列从头算一遍。

效果非常显著：对于长序列，KV Cache 能把推理速度提升数倍到数十倍，这也是 LLM 能以还算流畅的速度生成文本的重要原因之一。

---

## KV Cache 和 Context Window 的关系

这两个概念经常被混淆，但它们解决的是两个不同维度的问题：

| | Context Window | KV Cache |
|---|---|---|
| 解决的问题 | 模型能"看到"多少历史 | 推理计算效率 |
| 本质 | 输入文本的长度上限 | 计算中间结果的缓存 |
| 满了之后 | 必须截断或报错 | 显存不够，需要换出或重算 |
| 工程影响 | 决定 Agent 任务的可持续轮数 | 决定推理延迟和显存占用 |

一个直觉类比：

> Context Window 是你的"视野范围"——窗口之外你看不到。
> KV Cache 是你的"速记本"——把已经看过的内容记下来，不用每次重看。

两者共同决定了 LLM 推理的上限和效率。

---

## 对 Agent 设计的实际影响

理解了 context window 和 KV Cache，作为 Agent 工程师，有几个直接的设计决策：

**1. 设计 Memory 分层**

不要把所有历史都塞进 context。区分三类信息：
- **必须在 context 里的**：系统提示、当前任务目标、最近几轮的关键结果
- **按需检索的**：历史工具调用结果、参考文档（用 RAG 检索，而不是全部放进去）
- **可以丢弃的**：中间过程的冗余输出、已经被后续结果覆盖的旧结论

**2. 监控 context 使用量**

在 Agent 的每次推理前，记录当前 context 的 token 数。设置一个安全阈值（比如最大窗口的 70%），接近时触发压缩或摘要策略，而不是等到满了再报错。

**3. 理解"长 context 更贵"的根本原因**

调用 API 时，你会发现 context 越长，响应越慢、费用越高。这不是服务商在坑你，而是有物理原因：更长的 context 意味着更大的 KV Cache，需要占用更多 GPU 显存，服务商的硬件成本直接上升。同时，即使有缓存，每次推理仍需读取完整的 KV Cache，序列越长读取越慢。所以**压缩 context 不只是为了避免超出上限，也是在直接控制推理成本和延迟**。

---

## 小结

LLM 不是一个有持续记忆的智能体，而是一个每次都从零开始读取输入的函数。Context Window 定义了它能看到多远，KV Cache 让这个过程不那么慢。

理解这两点，你才能解释：
- 为什么 Agent 跑着跑着开始"忘事"
- 为什么长 context 的推理比短 context 慢得多
- 为什么 Memory 模块是 Agent 架构里不可省略的一层
