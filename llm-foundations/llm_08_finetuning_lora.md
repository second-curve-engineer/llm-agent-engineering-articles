# 预训练、微调、LoRA：三种"改造 LLM"的方式有什么区别？

---

你有没有遇到过这种需求：

业务上要用 LLM 处理特定领域的任务——比如按照公司内部的格式输出报告，或者理解某个垂直行业的专业术语。通用的 GPT/Claude 效果不稳定，总是"跑偏"。

这时候你会想：能不能让模型更专注于这个任务？能不能"教会"它公司的输出风格？

这就是**微调（Fine-tuning）**要解决的问题。

但微调不是一个方法，而是一类方法。从全量微调到 LoRA，成本从高到低，效果从强到弱（但差距没你想的那么大）。理解这三个层次，你才能在正确的场景做正确的选择。

---

## 第一层：预训练（Pre-training）

预训练是 LLM 能力的根基。

模型在海量文本上做 Next Token Prediction——给定前文，预测下一个词。训练数据通常是数万亿 Token 的互联网文本、书籍、代码等。这个过程需要数千块 GPU 跑几个月，成本从数百万到数亿美元不等。

预训练结束后，模型获得了对语言的基础理解：语法、事实知识、推理能力、代码能力……但它还不"听话"——让它写报告，它可能把后续的段落也一起生成出来，因为它学到的是"预测下一个词"，而不是"按指令完成任务"。

作为工程师，你几乎不会参与预训练——这是模型公司的工作，需要的资源远超一般团队的能力范围。

但理解预训练，你才能理解后续改造的"起点"是什么。

---

## 第二层：全量微调（Full Fine-tuning / SFT）

全量微调的做法是：拿预训练好的模型，在一个小得多的、针对特定任务的数据集上继续训练，**更新模型的所有参数**。

训练数据通常是"指令-回复"对：

```
指令：请将以下新闻摘要翻译成英文……
回复：Here is the English translation……

指令：请按照以下格式输出季度报告……
回复：Q3 2024 Report……
```

这个过程叫做 **SFT（Supervised Fine-Tuning，监督微调）**。OpenAI 把 GPT-3 变成能听指令的 InstructGPT，第一步就是 SFT。

全量微调的效果最好——所有参数都能针对新任务调整，模型能深度"适应"目标领域。

但代价也最高：

- **计算成本**：7B 模型全量微调，至少需要数块 A100，训练几个小时到几天
- **存储成本**：微调后的模型和原始模型一样大，7B 模型大约 14GB（fp16）
- **灾难性遗忘**：更新所有参数意味着模型可能"忘掉"一些原本掌握的通用能力，在非目标任务上性能下降

对大多数工程团队来说，全量微调 7B 模型还勉强可以接受，但 70B 以上基本不现实。

---

## 第三层：LoRA（低秩适配）

LoRA（Low-Rank Adaptation）是 2021 年微软提出的方案，解决了全量微调成本过高的问题。

它的核心思路是：**不更新原始模型的参数，而是在关键层旁边插入一对小矩阵，只训练这对小矩阵。**

### 为什么这样可行？

预训练模型的权重矩阵非常大（比如 4096×4096），但研究发现，微调时这些矩阵的**变化量**往往是低秩的——也就是说，真正有效的参数更新，可以用一个低秩矩阵来近似表达。

LoRA 的做法：对于原始权重矩阵 $W$，不直接更新它，而是在旁边加两个小矩阵 $A$（4096×r）和 $B$（r×4096），其中 $r$ 是一个很小的数（通常 4~64）。

推理时，实际使用的权重是 $W + AB$，其中 $AB$ 就是那对小矩阵的乘积。训练时只更新 $A$ 和 $B$，$W$ 完全冻结。

当 $r=8$ 时，一个 4096×4096 的矩阵（约 1600 万参数），被替换成两个小矩阵（4096×8 + 8×4096 = 约 6.5 万参数），参数量缩小到原来的 **0.4%**。

### LoRA 的实际效果

参数量大幅减少，效果损失有多少？

实践结论是：**在大多数任务上，LoRA 的效果和全量微调相差不大，而训练成本可以降低 10 倍以上。**

具体数字：
- 全量微调 7B 模型：需要 4~8 块 A100（80GB），约 14GB 显存仅用于模型本身
- LoRA 微调 7B 模型：单块消费级 GPU（24GB 显存）即可，甚至可以在 Mac M 系列芯片上运行

LoRA 还解决了"多任务共存"的问题：原始模型只需要保存一份，针对不同任务训练不同的 LoRA 权重，推理时按需加载。10 个任务 = 1 个基础模型 + 10 个小文件，而不是 10 个完整的大模型。

---

## QLoRA：进一步降低门槛

LoRA 还有一个增强版：**QLoRA（Quantized LoRA）**。

在 LoRA 的基础上，QLoRA 把原始模型的权重做量化（从 16bit 压缩到 4bit），进一步降低显存占用。

效果：**在单块消费级 GPU（24GB）上，可以微调 70B 参数的大模型。**

这在 QLoRA 出现之前是完全不可能的事。70B 全量微调需要至少 8 块 H100，而 QLoRA 把门槛降到了普通研究者和小团队可以承受的范围。

代价是：4bit 量化会带来一定的精度损失，但对大多数下游任务来说，这个损失在可接受范围内。

## 三种方式的对比

| | 预训练 | 全量微调（SFT） | LoRA / QLoRA |
|---|---|---|---|
| 目标 | 从零建立语言能力 | 适应特定任务 | 轻量级任务适配 |
| 训练数据 | 万亿 Token | 千~万条指令对 | 百~千条指令对 |
| 参数更新 | 全部 | 全部 | 0.1%~1% |
| 成本 | 极高（数百万美元） | 高（数块 A100） | 低（单卡可跑） |
| 效果 | 最强 | 强 | 接近全量微调 |
| 典型用途 | 模型公司 | 领域模型 | 任务适配、风格对齐 |

## 什么时候需要微调，什么时候不需要

这是最重要的判断——很多时候你根本不需要微调。

**先做 Prompt Engineering**

在考虑微调之前，先认真优化 prompt。加详细的系统提示、给几个示例（Few-shot）、加结构化输出约束……很多"模型跑偏"的问题，是 prompt 没写好，而不是模型能力不够。Prompt Engineering 的成本几乎为零，先把这条路走完。

**再考虑 RAG**

如果问题是"模型不知道业务相关的知识"，通常 RAG 比微调更合适。RAG 把知识存在外部数据库里，随时更新；微调把知识烧进模型参数里，更新需要重新训练。业务知识频繁变化的场景，RAG 的维护成本更低。

**再考虑微调**

真正需要微调的场景是：

- **输出风格/格式对齐**：需要模型严格按照某种特定格式输出，prompt 控制效果不稳定
- **领域术语理解**：大量行业专有名词，通用模型理解错误率高
- **推理效率**：把复杂的 few-shot prompt 压缩进模型，减少每次推理的 Token 消耗
- **隐私/合规**：不能把数据发给第三方 API，必须本地部署自己的模型

这些场景里，先用 LoRA，除非有明确的证据说明全量微调效果显著更好。

## 对 Agent 设计的实际影响

**1. 微调不是银弹**

很多团队遇到 Agent 效果不好，第一反应是"要不要微调一下"。但微调能解决的，是"模型的能力和风格对齐问题"，不能解决"架构设计问题"。如果 Agent 的工具调用逻辑有漏洞、Memory 设计不合理、prompt 结构混乱，微调只是在问题上贴创可贴。先把架构理清楚，再考虑微调。

**2. 微调数据的质量比数量重要**

LoRA 微调 100 条高质量的、覆盖边界 case 的数据，往往比 10000 条质量参差不齐的数据效果更好。构建微调数据集，要像写测试用例一样认真：覆盖正常输入、异常输入、边界情况，而不是简单地把业务数据堆进去。

**3. 微调后要做系统评估**

模型微调之后，不能只看目标任务的效果，还要回归测试通用能力。灾难性遗忘是真实存在的——微调让模型"专注"的同时，可能悄悄损坏了其他能力。这也是 Eval 体系重要的原因之一。

---

## 小结

预训练、全量微调、LoRA 是三个成本从高到低、适用场景不同的"改造 LLM"的层次：

- **预训练**：从零建立能力，模型公司的工作，工程师不参与
- **全量微调**：效果最强，成本最高，适合有算力资源的领域模型建设
- **LoRA/QLoRA**：轻量、低成本、效果接近全量微调，是工程团队做任务适配的首选

在考虑微调之前，先问自己：Prompt Engineering 和 RAG 能解决吗？大多数情况下，答案是能。

下一篇是 LLM 基础系列的最后一篇：**RLHF。** 微调让模型"会做任务"，但 RLHF 解决的是另一个问题——让模型"听话"，让它的输出符合人类的偏好和价值观。这也是 ChatGPT 能成为现象级产品的关键一步。
