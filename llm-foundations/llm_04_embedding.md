# Embedding：让 LLM "理解"语义的向量空间

---

上一篇讲了 Token——文本进入 LLM 之前，会被切成一段一段的 Token，每个 Token 对应一个数字 ID。

但数字 ID 本身没有任何语义信息。ID=1024 和 ID=1025，模型无法从这两个数字判断它们对应的词是相近还是相反。

要让模型真正"理解"语言，还需要下一步：**把每个 Token ID 转换成一个向量。这个向量，就是 Embedding。**

---

## 从 ID 到向量：一次语义的编码

Embedding 的核心思想是：**用一个高维向量来表示一个 Token，向量的每一个维度都编码了这个 Token 的某种语义特征。**

维度通常很高，从几百到几千不等。GPT-4 的 Embedding 维度大约是 12288 维。你可以把它理解成一个有 12288 个坐标轴的空间，每个 Token 在这个空间里有一个位置。

这个位置不是随机分配的，而是在训练过程中学出来的。模型通过大量文本，学会了把语义相近的词放在空间里相近的位置，语义相反的词放在相反的方向。

一个经典的例子：

> `向量(国王) - 向量(男人) + 向量(女人) ≈ 向量(女王)`

这不是人工设定的规则，而是模型从文本中自然学到的语义结构。"国王"和"女王"的关系，和"男人"和"女人"的关系，在向量空间里呈现出一致的方向和距离。

这就是 Embedding 的力量：它把离散的符号，变成了连续空间里可以计算的点。语义相似度，变成了向量之间的距离。

---

## 两种 Embedding：Token Embedding 和句子 Embedding

在实际使用中，Embedding 有两种常见的形态，用途完全不同。

**Token Embedding**

这是 LLM 内部用的。每个 Token 在进入 Transformer 之前，都会被转换成一个向量，作为模型计算的输入。整个序列就变成了一个向量矩阵，模型在这个矩阵上做 Attention 计算（下一篇会详细讲）。

这个过程对外不可见，是模型内部的表示层。

**句子 / 文本 Embedding**

这是对外暴露的 API 接口（比如 OpenAI 的 `text-embedding-3-small`）。输入一段文本，输出一个固定维度的向量，代表这段文本的整体语义。

这个向量可以用来做：
- **语义相似度计算**：两段文本的向量越接近，语义越相似
- **向量检索**：把大量文本都转成向量存起来，查询时用问题的向量去找最相近的文本
- **分类、聚类**：把文本映射到向量空间，用传统机器学习方法处理

RAG（检索增强生成）的底层，就是文本 Embedding + 向量检索。

---

## RAG 里的 Embedding：工程师最常接触的场景

如果你在做 Agent，RAG 几乎是绕不开的模块。而 RAG 的核心链路，就是 Embedding 在工程里的典型应用：

```
离线阶段：
文档 → 切片 → Embedding 模型 → 向量 → 存入向量数据库

在线阶段：
用户问题 → Embedding 模型 → 查询向量 → 向量数据库检索 → 返回最相似的文档片段 → 拼入 context → LLM 生成回答
```

这个流程里有几个工程细节值得注意：

**离线和在线必须用同一个 Embedding 模型**

向量是模型学出来的，不同模型的向量空间完全不同。用 A 模型建索引，用 B 模型查询，结果会是随机的——因为它们生活在两个不同的空间里，距离没有可比性。这个错误很隐蔽，因为程序不会报错，只是检索结果很差。

**相似不等于相关**

向量检索找的是语义最相似的文本，但"相似"和"能回答这个问题"不是同一回事。

举个例子：用户问"苹果公司的市值是多少"，向量检索可能返回"苹果公司发布了新产品"——这两段文本语义相似（都关于苹果公司），但后者根本没有市值信息。

这就是 RAG 里经常出现"检索到了但回答不准"的根本原因，不是 LLM 的问题，是检索策略的问题。解法包括：混合检索（向量检索 + 关键词检索）、重排序（Reranker）等。

**切片大小影响检索质量**

文档切片太大，向量会"稀释"——一个向量要表示太多内容，语义变得模糊，检索准确率下降。切片太小，单个片段缺乏上下文，即使检索到了，也可能答非所问。

通常的经验值是 300~800 Token 一个切片，具体要根据文档类型调整。

---

## Embedding 的局限性

理解 Embedding 的能力边界，同样重要。

**向量压缩了信息**

一段几百个 Token 的文本，被压缩成一个固定维度的向量。这个过程必然有信息损失。向量擅长捕捉整体语义和主题，但对细节、数字、逻辑关系的保留能力有限。

所以 Embedding 检索适合"找相关文档"，不适合"精确匹配内容"。后者用关键词检索更可靠。

**跨语言能力参差不齐**

不同的 Embedding 模型，对不同语言的支持程度差异很大。以英文语料为主训练的模型，处理中文时效果可能明显下降。做中文 RAG，要选择专门支持中文或多语言的 Embedding 模型，比如 BGE 系列（BAAI 出品）或 text-embedding-3 系列。

**维度越高不一定越好**

高维向量计算成本高、存储占用大。在实际工程里，要在检索质量和计算成本之间找平衡。OpenAI 的 `text-embedding-3-small`（1536 维）在大多数场景下效果已经足够，不必上 `text-embedding-3-large`（3072 维）。

## 对 Agent 设计的实际影响

**1. Embedding 模型是 RAG 的基础设施，要当成基础设施来管理**

一旦用某个 Embedding 模型建好了索引，切换模型意味着全量重建——所有文档要重新向量化，向量数据库要重新写入。这个成本很高，所以选型要慎重，不要随意跟进新模型。

**2. 区分"检索任务"和"生成任务"**

Embedding 检索解决的是"找到相关信息"，LLM 解决的是"基于信息生成回答"。这两步要分开优化。检索效果差，换 Embedding 模型或调整检索策略；生成效果差，优化 prompt 或换 LLM。混在一起调，永远找不到根本原因。

**3. 给检索结果加置信度判断**

向量检索总会返回结果，即使没有任何相关文档，它也会返回"最相似"的那几条。在 Agent 里，要对检索结果的相似度分数设置阈值：低于阈值的结果，宁可告诉 LLM"没有找到相关信息"，也不要把低质量内容拼进 context——后者会导致 LLM 产生幻觉。

---

## 小结

Embedding 把离散的 Token，转换成连续向量空间里的点。语义相近的词，在空间里位置相近；语义相反的词，方向相反。这让模型可以用数学运算来处理语义关系。

在工程里，Embedding 最直接的应用是 RAG 的检索环节。理解它的工作原理，你才能解释：
- 为什么 RAG 检索到了却回答不准
- 为什么离线和在线必须用同一个模型
- 为什么切片大小会影响检索质量

下一篇，我们进入 LLM 架构里最核心的机制：**Attention。** Token 被转成向量之后，模型怎么理解这些向量之间的关系？为什么"我吃了鱼，它很好吃"里的"它"，模型能知道指的是"鱼"而不是"我"？这就是 Attention 要回答的问题。
