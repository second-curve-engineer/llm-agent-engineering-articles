# LLM 怎么知道词的顺序？位置编码从 sin/cos 到 RoPE

---

上一篇讲 Transformer 架构时，有一个细节没有展开：

Attention 机制让每个 Token 和其他所有 Token 计算相似度，但这个计算本身**不感知顺序**。"我吃了鱼"和"鱼吃了我"，如果只看 Attention 的计算过程，模型拿到的是同一组向量，计算出来的结果完全一样。

但这两句话意思截然不同。

Transformer 是怎么解决这个问题的？答案是**位置编码（Positional Encoding）**：在 Token 向量里注入位置信息，让模型能区分同一个词出现在不同位置时的差异。

这个看似简单的"补丁"，背后的演进却直接决定了 LLM 能处理多长的上下文。

---

## 原始方案：sin/cos 位置编码

Transformer 的原始论文（"Attention is All You Need"，2017）用了一个优雅的数学方案：**用正弦和余弦函数生成位置向量，直接加到 Token 的 Embedding 上。**

具体来说，对于位置 $pos$、维度 $i$，位置编码是：

$$PE_{(pos, 2i)} = \sin\left(\frac{pos}{10000^{2i/d}}\right)$$
$$PE_{(pos, 2i+1)} = \cos\left(\frac{pos}{10000^{2i/d}}\right)$$

不用记这个公式，只需要理解它的两个关键性质：

**每个位置有唯一的编码**：不同位置生成的向量不同，模型可以区分位置 1、位置 2、位置 100。

**相邻位置的编码相似**：位置 5 和位置 6 的编码比位置 5 和位置 500 更接近，模型能感知位置的远近关系。

sin/cos 方案的优点是简单、不需要额外训练，而且理论上可以外推到训练时没见过的长度（因为是数学函数，不是查表）。

但它有一个致命问题：**实际外推效果很差。**

模型在训练时只见过一定长度的序列，当推理时序列超过训练长度，sin/cos 编码虽然能生成数值，但模型对这些"陌生"位置的处理能力急剧下降，输出质量快速崩坏。

---

## 进化：可学习的位置编码

GPT 系列早期用的是另一个思路：**把位置编码也当成参数，让模型在训练中学习。**

做法很简单：给每个位置分配一个可训练的向量，训练过程中和模型其他参数一起更新，最终学到一套"对当前任务最有用"的位置表示。

这个方案的效果往往比 sin/cos 更好，因为位置编码是针对实际训练数据优化出来的，而不是手工设计的数学公式。

但它的问题更明显：**完全无法外推。**

可学习位置编码是一张查找表，训练时有多少个位置，表里就有多少行。如果训练时最长序列是 2048 个 Token，表里就只有 2048 行。推理时序列超过 2048，直接越界——模型根本没有这个位置的编码，只能报错或截断。

这直接限制了早期 GPT 模型的上下文长度。GPT-2 的上下文窗口只有 1024 Token，GPT-3 扩展到 2048，都是被这个方案卡住的。

---

## 突破：RoPE（旋转位置编码）

RoPE（Rotary Position Embedding）是 2021 年提出的方案，现在几乎成了主流 LLM 的标配——Llama、Qwen、DeepSeek、Mistral 都在用。

RoPE 的核心思路和前两种方案都不同：**不是把位置信息加到 Token 向量上，而是把位置信息融入 Attention 的计算过程里。**

具体来说，RoPE 通过旋转矩阵，对 Q 和 K 向量按位置做旋转变换。旋转的角度由位置决定：位置越大，旋转越多。

这样做的结果是：当 Q 和 K 做点积计算注意力分数时，得到的分数自然地包含了两个 Token 之间的**相对位置关系**，而不是绝对位置。

举个直觉：两个词相差 3 个位置，不管它们在序列的开头还是结尾，Attention 分数里体现的"距离感"是一样的。

这个性质带来了一个关键优势：**RoPE 对长度外推的支持远好于前两种方案。**

---

## RoPE 为什么能支持长上下文

sin/cos 编码和可学习编码都是"绝对位置"的思路：每个 Token 有一个固定的位置 ID，模型记住的是"位置 5 的 Token 应该怎么处理"。

当序列超出训练长度，绝对位置 ID 进入了"没见过的区域"，模型完全懵了。

RoPE 记住的是**相对位置关系**：两个词相差多少步，它们之间的注意力分数就受到多大程度的"距离衰减"。这个关系在训练时已经学到，换一个绝对位置不影响相对关系的计算。

这就是为什么基于 RoPE 的模型，在做长上下文扩展时，比早期模型容易得多。

当然，RoPE 也不是无限外推。直接把 RoPE 模型推到训练长度的 10 倍，效果还是会下降。为了进一步扩展上下文，研究者在 RoPE 的基础上又开发了一系列技术：

**YaRN（Yet another RoPE extensioN）**：通过对旋转角度进行插值，让模型"平滑地"处理更长的序列，是目前最常用的长上下文扩展方案之一。Qwen 的长上下文版本就用了 YaRN。

**NTK-aware 插值**：从信号处理角度对 RoPE 的频率分布做调整，让高频分量和低频分量都能更好地外推。

这些方案本质上都是：在不重新训练的前提下，让 RoPE 能"伸展"到更长的序列。

---

## ALiBi：另一条路

除了 RoPE，还有一个方案值得了解：**ALiBi（Attention with Linear Biases）**。

ALiBi 的思路更激进：**完全不加位置编码，而是直接在 Attention 分数上加一个和距离成正比的惩罚项。**

两个 Token 相距越远，它们之间的注意力分数就越低，衰减量和距离成线性关系。

这个方案的优点是极其简洁，也有很好的长度外推能力——因为线性惩罚是一个统一的规则，不依赖于训练时的最大长度。BLOOM 模型用的就是 ALiBi。

但 ALiBi 也有缺点：强制的线性衰减会限制模型处理长距离依赖的能力。有些任务里，两个相距很远的词有强语义关联（比如长文档的首尾呼应），ALiBi 的惩罚机制会削弱这种关联。

实践中，RoPE 的综合表现更好，这也是为什么主流模型几乎都选了 RoPE 而不是 ALiBi。

## 对 Agent 设计的实际影响

位置编码听起来是底层架构细节，但它直接影响 Agent 工程里的几个决策：

**1. 不同模型的长上下文能力差异显著**

同样标称"支持 128K context"，基于 RoPE+YaRN 的模型和基于可学习位置编码的老模型，实际的长文本处理质量可能天差地别。选型时，不只看支持的 Token 数，还要看实际的长文本 benchmark 表现（比如 RULER、Needle-in-a-Haystack 测试）。

**2. 长 context 里的"注意力稀释"**

即使模型支持长 context，随着序列变长，每个 Token 能分配到的注意力权重在平均意义上会变小——因为要和更多词共享注意力。这意味着长 context 里的重要信息可能被稀释。在 Agent 设计里，不要把关键信息简单地扔进一个超长的 context，而要结合摘要、分块、RAG 等手段，帮模型聚焦在真正重要的内容上。

**3. 理解"上下文窗口"和"有效注意力范围"的差异**

模型支持 200K context，不代表它能均匀地理解这 200K 里的所有内容。位置编码的衰减特性（尤其是 ALiBi 的线性衰减）意味着越靠近当前 Token 的内容，模型关注度越高，远处的内容关注度递减。设计 Agent 的 Memory 策略时，要把最近的、最相关的内容放在 context 的靠近末尾的位置，而不是开头。

---

## 小结

Transformer 本身不感知词序，位置编码是给模型"装上感知顺序的眼睛"的机制。

- **sin/cos 编码**：数学公式生成，简洁但外推效果差
- **可学习位置编码**：训练出来的查表，效果好但完全无法外推
- **RoPE**：旋转变换融入 Attention，捕捉相对位置，主流方案
- **ALiBi**：线性距离惩罚，简洁但限制长距离依赖

理解位置编码，你才能真正理解"上下文窗口"这个概念的边界——不只是 Token 数上限，还有注意力能有效覆盖的实际范围。
