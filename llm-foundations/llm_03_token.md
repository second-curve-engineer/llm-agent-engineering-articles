# Token 是什么？Agent 工程师必须搞懂的基本单位

---

你有没有遇到过这些情况：

明明输入的文字不长，API 账单却比预期高了一截。或者模型在处理某个中文词的时候，表现出奇地差，换一种说法又好了。再或者，你设置了 max_tokens=500，但模型生成的内容看起来远不止 500 个字。

这些问题，根源都是同一个：**你对 Token 还没有直觉。**

Token 是 LLM 处理文本的最小单位，是计费的基准，也是模型"理解"语言的基础。搞懂它，你才能解释上面这些现象，也才能在工程上做出正确的决策。

---

## LLM 看到的不是文字，是 Token

人类读文章，是以字或词为单位来理解的。LLM 不是这样工作的。

在文本进入模型之前，会经过一个叫做 **Tokenization**（分词）的步骤：把原始文本切分成一段一段的 Token，再把每个 Token 转换成对应的数字 ID，模型实际处理的是这些数字序列。

Token 不等于字，也不等于词。它是介于两者之间的一种切分单位，由训练时使用的**词表**（Vocabulary）决定。

举个具体的例子，下面这句话：

> `I'm learning about tokenization`

被 GPT-4 的分词器切分后，变成：

> `I` `'m` ` learning` ` about` ` token` `ization`

一共 6 个 Token。注意：`tokenization` 被切成了 `token` 和 `ization` 两个 Token，因为这个词在词表里没有完整收录，但这两个部分分别收录了。

换成中文：

> `我在学习分词技术`

切分结果可能是：

> `我` `在` `学习` `分词` `技术`

也可能是：

> `我在` `学习` `分` `词技` `术`

具体怎么切，取决于模型使用的分词器和词表。

---

## BPE：主流分词算法的直觉

现在主流的 LLM 用的分词算法叫做 **BPE**（Byte Pair Encoding，字节对编码）。

它的核心思路是：**把高频出现的字符组合合并成一个 Token，直到达到词表大小上限。**

训练过程大概是这样的：

1. 初始状态，词表里只有单个字符（字母、汉字、标点……）
2. 统计所有相邻字符对出现的频率，把最高频的那对合并成一个新 Token
3. 重复这个过程，直到词表大小达到设定的上限（通常是几万到十几万个 Token）

结果是：高频词和高频词组会有自己独立的 Token，低频词和生僻词会被拆成更小的片段。

这解释了一个重要现象：**英文的 Token 效率通常高于中文。**

英文词汇量虽然大，但词根词缀组合规律强，高频组合容易被合并进词表。而中文字符数量庞大，常用汉字就有几千个，每个字都需要一个 Token 位置，词组合并的空间相对有限。粗略来说，同样的内容，中文需要的 Token 数往往是英文的 1.5~2 倍。

## Token 数如何影响你的日常工程决策

理解 Token 不是为了了解算法细节，而是因为它直接影响你的工程决策。

**计费**

OpenAI、Anthropic 等主流 API 都按 Token 计费，分输入（prompt tokens）和输出（completion tokens）两部分，输出通常比输入贵 2~3 倍。

你写的每一条系统提示、每一轮对话历史、每一段 RAG 检索结果，都在消耗输入 Token。模型生成的每一个字，都在消耗输出 Token。

在 Agent 场景下，如果你没有意识到 Token 消耗，很容易在循环调用中把成本放大几十倍。

**Context Window 的真实容量**

上一篇讲 Context Window 时，提到"128K tokens 大约是 10 万汉字"。但这个换算是粗略的，实际上中文内容的 Token 效率更低。

如果你的 Agent 主要处理中文内容，128K tokens 实际能容纳的汉字数量可能只有 6~7 万，而不是 10 万。在估算 context 上限时，需要把这个折扣考虑进去。

**max_tokens 控制的是 Token 数，不是字数**

`max_tokens=500` 意味着模型最多生成 500 个 Token，对应中文大约是 300~400 个汉字，不是 500 个字。如果你发现模型总是"说到一半截断了"，可能就是 max_tokens 设得太低了。

**生僻词、专业术语的理解问题**

词表里没有完整收录的词，会被拆成多个 Token。这本身没什么问题，但有时候拆法会破坏语义。

比如某些专业术语、人名、公司名，被拆开之后，模型对这个词的理解可能不如高频词准确。这是分词粒度带来的固有局限，不是 prompt 写得不好。遇到这种情况，可以在 prompt 里对这个词的含义做显式说明，帮模型"补全"语境。

## 怎么建立 Token 直觉

光说不够，要建立真正的 Token 直觉，最直接的方式是**亲手数一数**。

OpenAI 提供了一个在线工具 [Tokenizer](https://platform.openai.com/tokenizer)，可以把任意文本粘贴进去，直观看到它被切成了哪些 Token，以及总数是多少。建议你把自己的系统提示、常用的 prompt 模板都过一遍，对 Token 数量建立感知。

几个有用的经验数字：

| 内容类型 | 粗略换算 |
|---|---|
| 英文文本 | 1 Token ≈ 0.75 个单词，或 4 个字符 |
| 中文文本 | 1 个汉字 ≈ 1.5~2 个 Token |
| 代码 | 1 Token ≈ 3~4 个字符（关键字效率较高） |
| 空格、标点 | 通常各占 1 个 Token |

## 对 Agent 设计的实际影响

**1. 把 Token 计数放进 Agent 的监控里**

不要等到账单来了才发现超支。在 Agent 的每次 API 调用后，把 `usage.prompt_tokens` 和 `usage.completion_tokens` 记录下来，按任务统计总消耗。这既是成本控制的基础，也是发现"哪个步骤在浪费 Token"的手段。

**2. 系统提示要当成工程资产来管理**

系统提示每次调用都会完整输入，Token 数直接乘以调用次数就是成本。一个写得冗余的系统提示，在高频调用场景下可以产生可观的浪费。审查系统提示就像审查代码，要定期做"Token 瘦身"：去掉重复表达，压缩举例数量，用简洁的指令代替啰嗦的说明。

**3. 工具返回结果要控制长度**

Agent 调用工具后，返回结果会直接拼进 context。如果工具返回的是原始 API 响应，动辄几千 Token 的 JSON，很快就会撑爆 context，也会推高成本。工具层要做裁剪：只把 LLM 真正需要的字段传进去，其余的丢弃。

---

## 小结

Token 是 LLM 处理文本的最小单位，不是字，也不是词，而是由词表决定的切分片段。

理解 Token，你才能解释：
- 为什么 API 的计费和你的预期对不上
- 为什么中文内容消耗的 Token 比英文多
- 为什么 `max_tokens` 和实际字数之间有偏差
- 为什么有些生僻词模型处理得不好

下一篇，我们继续往深处走：Token 进入模型之后，第一件事是被转换成**向量**。这个向量，就是 Embedding——LLM 理解语义的真正起点。
