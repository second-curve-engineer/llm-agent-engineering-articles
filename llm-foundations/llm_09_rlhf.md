# GPT 是怎么变"听话"的？RLHF 原理工程师视角解读

---

上一篇讲了微调——通过在目标数据集上继续训练，让模型"会做"特定任务。

但"会做"不等于"听话"。

一个经过预训练的模型，给它一个问题，它可能会继续补全问题本身，而不是回答它。让它写邮件，它可能写出几十封格式各异的邮件，因为训练数据里就是这样的。让它拒绝有害请求，它可能完全不知道什么是"有害"。

**RLHF（Reinforcement Learning from Human Feedback，基于人类反馈的强化学习）** 解决的就是这个问题：不是让模型"更能干"，而是让模型的输出**符合人类的偏好和价值观**。

这是 ChatGPT 能成为现象级产品的关键一步，也是理解当前所有主流 LLM 行为模式的底层逻辑。

---

## 问题的本质：预训练模型不知道"好"是什么

预训练的目标是预测下一个词，没有任何关于"回答质量"的概念。模型学到的是"什么样的文本在统计上是合理的"，而不是"什么样的回答对用户有帮助"。

这导致了几个典型问题：

**指令遵循差**：让模型"用三句话总结"，它可能写了十句话，因为更长的文章在训练数据里更常见。

**有害内容**：互联网文本包含大量有害内容，预训练模型会原样学到，并在适当的 prompt 下输出。

**过度自信**：模型在不确定的话题上也会给出确定性的回答，因为训练数据里的文本通常不带有"我不确定"这样的表达。

**奉承倾向**：模型会迎合用户的观点，即使用户的观点是错误的，因为"认同"在对话数据里往往获得更高评分。

SFT 能缓解这些问题，但有上限——你能构建的指令数据集，永远无法穷举所有情况。RLHF 的思路更根本：**直接让人类的偏好判断参与训练过程，让模型学会"什么是好的输出"。**

---

## RLHF 的三个步骤

### 第一步：SFT（监督微调）

和上一篇讲的一样，先用高质量的"指令-回复"数据对预训练模型做一轮监督微调，让模型基本学会"按指令回答"的格式。

这是 RLHF 的起点，SFT 模型是后续步骤的基础。

### 第二步：训练奖励模型（Reward Model）

这是 RLHF 最关键、也最有创意的一步。

做法是：让 SFT 模型对同一个问题生成多个不同的回答，然后让**人类标注员**对这些回答进行排序，判断哪个回答更好。

比如对于问题"解释一下量子纠缠"，模型生成了回答 A、B、C 三个版本。标注员看完之后，判断 B > A > C（B 最好，C 最差）。

收集大量这样的排序数据之后，用这些数据训练一个**奖励模型（Reward Model，RM）**。奖励模型的任务是：给定一个问题和一个回答，输出一个分数，表示这个回答有多"好"。

训练完成后，奖励模型就成了一个自动化的"人类偏好评估器"——它学到了人类对"好回答"的判断标准，可以对任意回答打分，不再需要人类实时介入。

### 第三步：强化学习优化（PPO）

有了奖励模型，就可以用强化学习来优化 SFT 模型了。

具体算法是 **PPO（Proximal Policy Optimization，近端策略优化）**。把 SFT 模型当作"策略"（Policy），让它生成回答，用奖励模型给回答打分，然后用 PPO 更新模型参数，让模型倾向于生成高分回答。

这个过程反复迭代：模型生成回答 → 奖励模型打分 → PPO 更新模型 → 模型生成更好的回答 → ……

但有一个重要的约束：PPO 在更新模型时，会限制新模型和原始 SFT 模型之间的差距不能太大（这就是"Proximal"的含义）。这个约束防止模型为了追求高分而走极端——比如学会"骗过"奖励模型，输出一些奖励模型打高分但实际上没用的内容（这叫 **reward hacking**）。

## DPO：更简洁的替代方案

PPO 算法复杂，训练不稳定，对超参数非常敏感，工程实现成本很高。

2023 年，斯坦福提出了 **DPO（Direct Preference Optimization，直接偏好优化）**，跳过了奖励模型和强化学习，直接用人类排序数据来优化模型。

DPO 的思路是：给定同一个问题的"好回答"和"坏回答"对，直接训练模型提高好回答的概率、降低坏回答的概率。

数学上可以证明，DPO 等价于用一种特殊形式的奖励模型做 RLHF，但工程上简单得多——不需要单独训练奖励模型，不需要跑 PPO，直接用监督学习的方式就能完成。

实践结论：**DPO 在大多数任务上效果和 PPO-based RLHF 相当，但训练成本低很多，稳定性更好。** 现在很多开源模型的对齐阶段用的都是 DPO 或其变体（如 ORPO、SimPO）。

## RLHF 解决的不是能力问题，是对齐问题

理解 RLHF 有一个最重要的认知前提：

**RLHF 不是让模型变得更聪明，而是让模型的行为和人类期望对齐。**

模型的知识和推理能力来自预训练。RLHF 改变的是模型的"态度"：
- 遇到有害请求，应该拒绝
- 遇到不确定的问题，应该表达不确定性
- 遇到用户错误的观点，应该温和地纠正
- 给出答案时，应该有用、无害、诚实

这也解释了一些现象：

**为什么 LLM 有时候"过度谨慎"？** RLHF 训练时，标注员对安全性的偏好被放大，导致模型在边界情况下倾向于过度拒绝，而不是冒险回答。

**为什么 LLM 有时候"过度迎合"？** 奖励模型学到的"好回答"标准，可能包含了"标注员喜欢被认同"的偏差，导致模型有讨好倾向（sycophancy）。

**为什么不同的 LLM 在相同问题上态度不同？** RLHF 的训练数据和标注标准不同，导致不同模型的"价值观"有差异。Claude 的 Constitutional AI、GPT 的 InstructGPT、Llama 的 RLHF 方案，都有各自的偏好权衡。

---

## Constitutional AI：Anthropic 的改进方案

Anthropic（Claude 的开发者）提出了 **Constitutional AI（CAI）**，在 RLHF 的基础上做了一个重要改进：

不完全依赖人类标注员来判断"好坏"，而是先制定一套明确的**原则（Constitution）**，然后用模型本身来判断回答是否符合这些原则，再用这些判断来做强化学习。

这个方案的优点：
- 减少对大量人类标注的依赖，降低成本
- 原则是显式的、可审查的，而不是隐含在标注员的主观判断里
- 模型的行为更可预测，更容易解释

Claude 的相对稳定的价值观和对边界情况的处理，很大程度上来自 CAI 的训练方式。

## 对 Agent 设计的实际影响

**1. 理解模型的"拒绝"行为**

RLHF 训练出来的模型，对某些话题会自动触发拒绝机制。在 Agent 设计里，如果你的任务涉及这些边界，不要试图用 trick 绕过，而是重新审视任务定义。如果业务确实需要模型处理某类敏感内容（比如医疗信息、法律分析），考虑用经过特殊对齐的专业模型，而不是通用模型强行 hack。

**2. 警惕奉承倾向对 Agent 结果的影响**

RLHF 带来的 sycophancy 问题，在 Agent 场景里会被放大。如果 Agent 的前一步输出了错误的中间结果，后续步骤的模型调用可能因为"迎合"而接受这个错误，而不是纠正它。在关键节点加入独立的验证步骤（不依赖模型自我判断），比期待模型自我纠错更可靠。

**3. 不同模型的"价值观"差异影响任务分配**

Claude 在安全性和诚实性上的训练权重更高，在某些边界任务上更容易拒绝；GPT 系列在指令遵循上的训练更激进；开源模型（Llama、Qwen）的对齐程度参差不齐。选择模型时，不只看能力 benchmark，还要了解它的对齐方向，把合适的任务交给合适的模型。

**4. RLHF 之后的模型不适合直接做 Base 继续微调**

RLHF 对齐之后的模型（Chat 版本），如果你想在上面做 LoRA 微调，要小心——针对特定任务的微调可能会损坏 RLHF 带来的安全对齐。如果需要既有任务能力又要保持安全性，考虑从 Base 模型（未经 RLHF）微调，然后加上自己的安全层；或者用专门设计来在 Chat 模型上微调的方案。

## 小结

RLHF 是让 LLM 从"能干"变成"听话"的关键机制：

- **SFT** 让模型学会按指令格式回答
- **奖励模型** 把人类的偏好判断自动化
- **PPO/DPO** 用奖励信号优化模型，让它倾向于生成人类认为"好"的回答

理解 RLHF，你才能解释：
- 为什么 LLM 有时候过度谨慎或过度迎合
- 为什么不同模型在相同问题上态度不同
- 为什么微调 Chat 模型要比微调 Base 模型更复杂
