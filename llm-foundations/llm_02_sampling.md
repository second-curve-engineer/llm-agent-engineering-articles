# 为什么同一个 prompt，LLM 每次给的答案都不一样？

## 采样策略：Temperature、Top-p、Top-k 详解

---

你有没有遇到过这种情况：

同一个 prompt，昨天跑出来的结果很好，今天再跑，答案变了。你没改任何东西，模型也没升级，但输出就是不一样。

或者反过来：你希望 Agent 每次执行相同的任务时，给出稳定一致的结果，但它就是会"随机发挥"。

这不是 bug，也不是模型不稳定。这是 LLM 的设计如此。

要理解为什么，你需要先明白一件事：**LLM 的输出本质上是一次概率采样，不是查表。**

---

## LLM 在做什么：输出一个概率分布

很多人以为 LLM 是这样工作的：输入一个问题，模型在某个巨大的"答案库"里查找最匹配的答案，然后返回给你。

实际上完全不是这样。

LLM 每次生成一个 token，做的事情是：根据当前的 context，计算词表里**每一个** token 出现在下一个位置的概率。词表通常有几万个 token，模型会给每个 token 打一个分数，然后把这些分数转换成概率分布——所有 token 的概率加起来等于 1。

然后，从这个概率分布里**采样**一个 token，作为输出。

注意这个词：**采样**。不是取最高概率的那个，而是按概率分布随机抽取。

这就是为什么同一个 prompt 每次输出都不同——每次采样的结果本来就不一定一样，就像掷一个不均匀的骰子，每面出现的概率不同，但每次都有机会掷出任意一面。

当然，概率高的 token 被采到的概率也高，所以输出不是完全随机的，而是"有倾向的随机"。

---

## Greedy Decoding：永远选最高概率

在讲采样策略之前，先看最极端的情况：如果我们**不采样**，永远选概率最高的那个 token，会怎样？

这叫 **Greedy Decoding**（贪婪解码）。

好处很明显：输出完全确定，同一个 prompt 永远给出同样的结果。

坏处也很明显：输出会变得刻板、重复。因为每一步都选"最安全"的选项，模型失去了探索其他合理路径的机会。生成长文本时，这种策略会让输出读起来越来越像流水账，缺乏变化和创造性。

所以大多数情况下，我们不用纯粹的贪婪解码，而是在确定性和随机性之间找一个平衡点。这就是采样策略要解决的问题。

---

## Temperature：控制随机程度

**Temperature** 是最常见的采样参数，也是影响最大的一个。

它的作用，是在采样之前对概率分布做一次"拉伸或压缩"：

**Temperature 越低（接近 0）**：概率分布变得更"尖"，高概率的 token 被进一步放大，低概率的 token 被进一步压缩。极限情况下退化为 Greedy Decoding，输出趋于确定。

**Temperature 越高（比如 1.5、2.0）**：概率分布变得更"平"，高低概率之间的差距缩小，原本不太可能被选中的 token 也有了更多机会。输出更随机、更有创意，但也更容易"跑偏"。

**Temperature = 1.0**：不做任何调整，使用模型原始输出的概率分布。

一个直觉类比：

> Temperature 就像一个"惊喜旋钮"。调低，输出越来越可预测；调高，输出越来越意想不到。

在 Agent 工程里，Temperature 的选择直接影响任务质量：

- 写代码、调用工具、做数据提取：用低 Temperature（0.0 ~ 0.3），你需要稳定、可重复的输出
- 写文案、做头脑风暴、生成多样化建议：用高 Temperature（0.7 ~ 1.2），你需要创意和变化
- 通用问答：0.5 ~ 0.7 是常用区间

---

## Top-k：只从前 k 个候选里采样

Temperature 调整的是整个概率分布的形状，但有时候我们还想更直接地控制：**只允许从概率最高的 k 个 token 里采样，其余的直接排除在外**。

这就是 **Top-k** 采样。

比如 Top-k=50，意味着每次采样只从概率最高的 50 个 token 里选，排名 51 之后的 token，无论有多少概率，都不会被选中。

Top-k 能有效防止模型输出概率极低的"离谱内容"。

但它有一个缺陷：k 是固定的，不管当前概率分布是宽还是窄，都从 50 个里选。

- 如果当前概率分布很"集中"（比如下一个词几乎确定是"的"），Top-50 里有大量无意义的候选凑数
- 如果当前概率分布很"分散"（多个词都合理），Top-50 可能又太窄，切掉了很多合理选项

这个问题，Top-p 解决得更优雅。

---

## Top-p（Nucleus Sampling）：动态候选池

**Top-p** 的逻辑不是"选前 k 个"，而是：**从概率最高的 token 开始累加，直到累积概率达到 p，取这个范围内的 token 作为候选池**。

比如 Top-p=0.9，模型会把候选 token 按概率从高到低排列，从第一个开始累加，直到累加到 0.9 为止。这个范围内的 token 就是候选池。

候选池大小是**动态的**：

- 概率分布集中时，少数几个 token 就能凑到 0.9，候选池小，输出更确定
- 概率分布分散时，需要很多 token 才能凑到 0.9，候选池大，输出更多样

Top-p 比 Top-k 更能适应不同场景的概率分布，这也是为什么大多数 API 默认用的是 Top-p 而不是 Top-k。

---

## 三个参数怎么配合使用

实际调用 API 时，Temperature、Top-k、Top-p 可以组合使用，执行顺序是：

```
原始概率分布
  → Temperature 调整分布形状
  → Top-k 过滤（如果设置了）
  → Top-p 过滤（如果设置了）
  → 从剩余候选池里采样
```

常见的组合建议：

| 场景 | Temperature | Top-p | 说明 |
|---|---|---|---|
| 代码生成 / 工具调用 | 0.0 ~ 0.2 | 0.9 | 要确定性，Temperature 优先调低 |
| 通用问答 | 0.5 ~ 0.7 | 0.9 | 平衡准确和自然 |
| 创意写作 | 0.8 ~ 1.2 | 0.95 | 鼓励多样性 |
| 结构化输出 / 数据提取 | 0.0 | — | 直接用 Greedy，要的就是唯一答案 |

一个实用原则：**先调 Temperature，它的影响最大；Top-p 作为辅助过滤，通常保持 0.9 左右不动。** Top-k 在大多数场景下不需要单独设置。

## 对 Agent 设计的实际影响

**1. 不同节点用不同参数**

Agent 是多个步骤串联的流程，规划、工具调用、总结，对随机性的需求完全不同。不要用同一套参数跑所有节点——工具调用用 Temperature=0，规划和总结可以稍高一些。

**2. 可复现性的边界**

如果你的 Agent 需要可复现（比如测试、回归对比），Temperature 设为 0 是最简单的方法。但要注意：即使 Temperature=0，不同的模型版本、不同的硬件和并发负载，也可能导致输出细微差异。想要真正可复现，还需要固定模型版本，并配合 Eval 体系来验证——这是后面单独一篇要讲的内容。

**3. 偶发性错误的根源**

有时候明明 prompt 写得很清楚，模型还是会偶尔输出格式不对、内容偏离的结果。这不一定是 prompt 的问题，也可能是采样走到了低概率路径。降低 Temperature 能减少偶发错误，但更根本的解法是加结构化输出约束（比如要求 JSON schema），直接限制候选 token 的范围，从源头堵住偏离的可能。

---

## 小结

LLM 的输出不是查表，是概率采样。每次生成一个 token，模型在整个词表上维护一个概率分布，然后按分布随机抽取。

- **Temperature** 控制分布的"尖锐"程度，决定输出的随机性
- **Top-k** 限制候选 token 的绝对数量
- **Top-p** 动态调整候选池大小，适应不同的分布形状

理解这三个参数，你才能解释：
- 为什么同一个 prompt 每次输出不同
- 为什么代码生成要用低 Temperature
- 为什么 Agent 会偶发性地"随机发挥"，以及怎么应对
