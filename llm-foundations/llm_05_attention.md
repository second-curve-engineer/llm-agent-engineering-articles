# 看懂 Attention，你才真正懂 LLM

---

上一篇讲了 Embedding——文本被切成 Token，每个 Token 被转换成一个向量。现在模型手里有了一排向量，比如"我吃了鱼，它很好吃"这句话，就变成了 8 个向量。

但向量和向量之间是相互独立的。模型怎么知道"它"指的是"鱼"而不是"我"？

这就是 **Attention** 要解决的问题：**让每个 Token 在处理自身的时候，能够"看到"序列里其他 Token 的信息，并根据相关性决定参考谁、参考多少。**

Attention 是 Transformer 架构的核心，也是理解 LLM 一切行为的底层机制。搞懂它，你才真正懂 LLM 在做什么。

---

## 一个直觉：查询、索引、内容

Attention 的计算涉及三个矩阵：**Q（Query）、K（Key）、V（Value）**。这三个名字来自信息检索领域，用这个类比来理解最直接：

想象你在用搜索引擎：
- 你输入的**搜索词**，就是 **Query（查询）**
- 数据库里每篇文章的**标题和标签**，就是 **Key（索引）**
- 每篇文章的**实际内容**，就是 **Value（内容）**

搜索的过程是：用你的 Query 去匹配所有文章的 Key，计算相关度，然后按相关度加权返回 Value。

Attention 做的事情完全一样：

对序列里的**每一个 Token**，用它的 Q 去和所有 Token 的 K 计算相似度，得到一组注意力分数，再用这组分数对所有 Token 的 V 做加权求和，得到这个 Token 的新表示。

用"我吃了鱼，它很好吃"来说：

当模型处理"它"这个 Token 时：
- "它"的 Q 去和序列里所有词的 K 计算相似度
- "鱼"的 K 和"它"的 Q 相似度最高（因为"它"在语义上更可能指代名词）
- 最终"它"的新表示，主要来自"鱼"的 V，也就是说"它"的语义被"鱼"的语义所塑造

这就是 Attention 的本质：**动态地、有选择地聚合序列中其他位置的信息。**

---

## Self-Attention：每个词都和所有词交互

上面描述的，就是 **Self-Attention**（自注意力）。"Self"的意思是：Q、K、V 都来自同一个序列，每个 Token 既是"查询者"，也是"被查询者"。

Self-Attention 的计算步骤：

**第一步：生成 Q、K、V**

对输入的每个 Token 向量，分别乘以三个学习到的权重矩阵（$W_Q$、$W_K$、$W_V$），得到这个 Token 的 Q、K、V 向量。

**第二步：计算注意力分数**

用每个 Token 的 Q，和所有 Token 的 K 做点积，得到一组原始分数。做一次 Softmax，把分数归一化成概率分布——这就是注意力权重。

**第三步：加权求和**

用注意力权重对所有 Token 的 V 做加权求和，得到每个 Token 的新向量表示。

这个新向量，已经融入了序列中其他词的信息。"它"的新向量，不再只是"它"本身的含义，而是带着"鱼"的语义的"它"。

---

## 为什么需要"多头"：Multi-Head Attention

一个 Attention 头，只能捕捉一种"关注模式"。比如上面的例子里，一个头捕捉了"代词-名词"的指代关系。

但语言里的关系不只这一种：主谓关系、修饰关系、因果关系……不同类型的语义关联，需要不同的"关注视角"来捕捉。

**Multi-Head Attention**（多头注意力）的思路是：并行运行多个 Attention 头，每个头用不同的 $W_Q$、$W_K$、$W_V$，学习捕捉不同类型的关系。最后把所有头的输出拼接起来，再做一次线性变换，得到最终表示。

GPT-4 有 128 个注意力头。128 个头同时工作，从 128 个不同的视角分析序列里的关系，最终汇总成一个丰富的表示。

这也解释了为什么大模型能理解复杂的语义关系——不是靠某个单一的机制，而是靠大量并行的"视角"叠加。

---

## Attention 的计算代价

Attention 有一个固有的代价：**计算量随序列长度的平方增长。**

序列长度为 N 时，每个 Token 都要和其他所有 Token 计算相似度，总计算量是 $N \times N = N^2$。

序列长度翻倍，计算量变成 4 倍。这就是为什么长 context 推理比短 context 慢得多、贵得多——不是线性增长，是平方增长。

这也是前面提到的 KV Cache 如此重要的原因：在生成阶段缓存已经计算过的 K 和 V，避免重复计算，是对这个平方代价最直接的优化。

---

## Causal Attention：GPT 为什么只看左边

Transformer 有两种主要形态：

- **Encoder（BERT 类）**：每个 Token 可以看到序列里所有其他 Token，包括右边的词。适合理解任务（分类、问答、语义匹配）。

- **Decoder（GPT 类）**：每个 Token 只能看到它左边的词，不能看右边。这叫 **Causal Attention**（因果注意力）或 Masked Attention。

为什么 GPT 要做这个限制？

因为 GPT 的任务是"预测下一个词"。训练时，如果让模型"偷看"右边的词，它就直接抄答案了，学不到任何东西。把右边遮住，强迫模型只根据左边的历史来预测，才能真正学会语言的规律。

推理时同样遵循这个规则：模型生成第 N 个词时，只能参考前 N-1 个词，这和训练时的设定保持一致。

这个设计决定了 GPT 类模型的核心能力是**生成**，而不是双向理解。

---

## Attention 告诉你：LLM 的"理解"是什么

理解了 Attention，你才能真正理解 LLM 的"理解"是什么——不是像人类一样有语义直觉，而是：

**通过大量参数学会的、在向量空间里的、基于相似度的动态信息聚合。**

这个理解有几个直接的工程含义：

**1. 上下文位置影响注意力分配**

研究发现，LLM 对 context 开头和结尾的内容注意力更高，对中间部分的注意力相对较低——这就是"Lost in the Middle"现象。所以在设计 prompt 时，最重要的指令和信息，放在开头或结尾，不要埋在中间。

**2. 重复出现的信息获得更多注意力**

在 context 里多次提及的关键信息，往往会获得更高的注意力权重，模型对它的"印象"更深。如果你想强调某个约束或重要信息，可以在 prompt 的不同位置重复强调，而不是只写一次。

**3. Attention 是可解释的**

相比神经网络的其他部分，Attention 权重是相对可观察的。有工具可以可视化某个 Token 在生成时"最关注"哪些输入词。在 Agent 调试中，如果模型行为异常，Attention 可视化是一个有价值的分析工具，能帮你判断模型是否在关注正确的信息。

---

## 小结

Attention 是 LLM 理解上下文关系的核心机制。它让每个 Token 在更新自身表示时，能够动态地参考序列里其他位置的信息。

- **Q/K/V** 的直觉：查询-索引-内容，用相似度决定参考谁
- **Self-Attention**：序列内部每个词和所有词交互，捕捉语义关系
- **Multi-Head Attention**：多个头并行，从不同视角捕捉不同类型的关系
- **Causal Attention**：GPT 只看左边，保证生成任务的训练一致性
- **平方复杂度**：序列越长计算量越大，这是长 context 推理慢的根本原因
